# Battle Logic (對戰相關)

## Core rule: fix single-player AND PvP together

Every battle mechanic exists **twice**, independently implemented: `pokemon_battle.html`/`public/single.html` (client-only, CPU opponent) and `server.js` (server-authoritative) + `public/pvp.html` (thin render client). They are NOT sharing code. A fix in one almost never automatically applies to the other — treat every battle-logic change as two separate patches that happen to be conceptually identical. This has been the single most common source of "works in single-player but not PvP" bugs across every session so far.

After editing `pokemon_battle.html`, sync: `cp pokemon_battle.html public/single.html && diff` to confirm.

## Turn flow (single-player)

```
playerAttack(i) / playerSwitch(i) / playerStandby()
  └─ doAttack(player→cpu) → cpuAttack()
       └─ cpuUseTrainers()  [CPU plays trainer cards from hand first]
          └─ doAttack(cpu→player)
               └─ handleKO(side, cb) if HP ≤ 0
                    └─ showResult() if that side's whole team is wiped, else cb()
```
`G.busy=true` disables player input during animations/CPU turn; resets via the various `endTurn`-style callbacks.

Switching (`playerSwitch` → confirm modal → `executeSwitch`) **ends the turn** by default (×0.8 damage guard applied to the switched-in mon, `G.playerSwitchGuard`) — except when `G.freeSwitch` is set (granted by the 撤退背心/retreat-vest item card), in which case it skips the confirm modal and does NOT end the turn. `G.freeSwitch` was a previously-dead unused field from an older design (换人命令 card) — repurposed rather than adding a new one; if you see other suspiciously-unused `G.*` fields, check git history before assuming they're safe to reuse.

## Turn flow (PvP) — `server.js`

WS message types: `attack`, `switch`, `standby`, `ko_switch`, `use_trainer`, `discard`, `discard_trade`. Handler is one big `if (type === '...')` chain inside `handleMessage(ws, msg)`. `G.turn` (`'p1'|'p2'`) is the authoritative whose-turn-is-it flag; `G.pendingKOSwitch` (role or null) gates forced replacement after a KO — while set, most other message types are rejected.

`G.pendingKOSwitchQueue` (array) — built for 搏命/sacrifice, reused for the double-KO fix — lets **both** sides need a forced replacement in sequence: `pendingKOSwitch` is set to the first side, and once their `ko_switch` resolves, the `ko_switch` handler shifts the next role off the queue automatically. Whether `G.turn` changes during this depends on *why* both died (see Double-KO below) — don't assume it's always the same.

## Trainer cards

`TRAINERS` array (`{id, name, cat:'item'|'supporter'|'stadium', desc}`) + `applyTrainer(card, side/role, ...)` switch statement handle the simple, single-effect cards (heal, buffs, status infliction, orbs, stadiums, energy). Items: unlimited per turn. Supporters: **1/turn, no per-stage/per-game cap** (the old "2/stage(single) or 2/game(PvP)" limit was removed 2026-07-02 at the user's request — only the `SuppUsed`-per-turn flag remains; `SuppStageUsed` still increments but nothing reads it as a gate anymore), **only appear in the opening 3-card hand** (never drawn later — post-opening draws are items-only).

**傷藥 (potion) is now 3 item cards, not 4** (`potion-s`/小傷藥 removed entirely 2026-07-18, alongside a matching global rebalance — see below): `potion-m`/`potion-l`/`potion-xl` healing 40/60/80 respectively, one shared `case 'potion-m': case 'potion-l': case 'potion-xl':` block in `applyTrainer` that looks the heal amount up from a small `{...}[card.id]` map rather than near-duplicate cases. If you add a new tier, extend that map, don't add a new switch case. Single-player's CPU heal-selection AI (`useCard('nurse') || useCard(...)` inside `cpuUseTrainers()`) tries all remaining tiers best-to-worst (`potion-xl` → `potion-l` → `potion-m`) after `nurse` — update that chain too if tiers change, it's a separate list from the switch case. 復活藥/revive also dropped from 80→40 HP in the same pass (both its `desc` string and the actual `deck[di].cur = 40` in `applyTrainer`'s `case 'revive':`).

**2026-07-18 global rebalance**: user asked for the damage bump to be offset by a healing nerf — every move's `dmg` field (both engines, all 152 Pokémon, 513 damage-bearing move slots — support moves have no `dmg` field and were untouched) got **+30** via a one-shot global regex (`dmg:(\d+)` → `dmg:` + n+30, safe because that exact literal pattern only ever appears in move objects, verified by grepping for `dmg:` outside `attacks:[` context before running it — the runtime damage-calc code always uses `let/const dmg =`, never the `dmg:` object-literal shorthand). Every **flat-number heal card** got **-40** (potion-m/l/xl, revive — see above); 小傷藥/potion-s was removed outright rather than reduced, per explicit instruction. **Percentage-based heals were deliberately left alone** (poison-heal 1/8 max HP, water-absorb 1/4, tidal-heal 30%, `selfHeal` moves, roost 50%, hot-springs' flat 15/turn, curse-wave's 20 cap) — a flat -40 doesn't apply cleanly to a percentage, and the smaller flat amounts (15/20) would go negative; only card-based, single-use, large-flat-number heals were in scope. If asked to rebalance heals again, this scoping precedent (potion-family + revive only, not passive/percentage effects) is what was used — confirm with the user if a future request seems ambiguous about which heal effects it should touch.

**Mega 稜鏡塔 (`stadium-mega-prism`)** is a normal stadium card (same generic activation case as every other stadium — `G.activeStadium = card`) whose actual per-turn effect lives outside `applyTrainer`, in the same turn-start hooks used for Mega-energy accumulation — see battle-logic skill's Mega Evolution section above.

Two cards are too complex for the generic `applyTrainer` switch and are special-cased directly in `useTrainer()`/PvP's `use_trainer` handler instead:
- **搏命 (sacrifice)**: both current actives die simultaneously. Single-player: `resolveMutualKO()`. PvP: inline in the `use_trainer` handler, sets up `pendingKOSwitch`+`pendingKOSwitchQueue`. **Both engines end the card player's turn once both sides replace** (fixed 2026-07-04 — originally neither did, letting the card's user act again immediately after both replacements were picked, reported by the user as "太強了"; single-player's final callback now calls `setTimeout(cpuAttack, 700)` instead of just clearing `G.busy`, and PvP now sets `G.turn = op` in the sacrifice block, mirroring the already-correct double-KO-from-attack case a few lines below it in `server.js`). If you touch this again, treat "does the turn actually end" as the thing to verify first — it's easy to wire up the replacement-picking UI correctly while silently forgetting the handoff.
- **瘋狂博士 (mad-scientist)**: 2-step selection UI (pick own alive mon → pick opponent's fainted mon) → `transformPokemon()` mutates the chosen mon's identity **in place** (same object reference — this has UI implications, see ui-rendering skill's sprite section). PvP validates both target indices server-side before applying (never trusts client-computed results); client sends `{type:'use_trainer', handIdx, targetOwnIdx, targetEnemyIdx}`. **HP handling changed 2026-07-15**: originally carried over the transforming Pokémon's *current HP percentage* onto the new max HP (`mine.cur = round(target.hp * (mine.cur/mine.hp))`) — user feedback was that this made the card feel bad exactly when you'd want it most (using it while nearly dead just produced a nearly-dead copy of the new Pokémon, solving nothing). Changed to always fully heal on transform (`mine.cur = target.hp`), matching Mega Evolution's existing "HP全滿" precedent in this same game — status was already cleared on transform before this change, so transform now matches Mega's "full heal + status clear" behavior exactly. The "can only target the opponent's *fainted* Pokémon" restriction was deliberately left alone (considered intentional strategic depth — save the card for after someone's died — not a flaw) even though it makes the card unusable before any KO has happened; only the HP-percentage part was judged to be the actual problem.

CPU AI (`cpuUseTrainers()`, single-player only — PvP has no AI) deliberately does **not** know how to play 搏命 or 瘋狂博士 (too complex to integrate safely into the synchronous turn flow) — they'll sit unused in CPU's hand if drawn. This is intentional, not a gap to fill without being asked.

**Two follow-up bugs found right after the 2026-07-06 card pass shipped, both from the same category of mistake — a card/mechanic added to the data array but not wired into every place that needs to know about it:**
1. **New stadium cards need adding to the *generic activation* switch case too, not just the `TRAINERS` data array.** `applyTrainer`'s stadium-activation case is a hand-enumerated list of ids (`case 'stadium-training': case 'stadium-spring': ... case 'stadium-mega-prism': { G.activeStadium = card; ... }`) — adding a new `cat:'stadium'` entry to `TRAINERS` does *not* automatically make it fall into this case; forgetting to list the new id here means the card gets consumed from hand with zero effect (`G.activeStadium` never gets set), reported by the user as "陷阱卡發動了沒效果". Both new trap stadiums (`stadium-spikes`, `stadium-toxic-field`) had this exact gap in both engines.
2. **交換器 (switcher) forces an opponent switch but never called `triggerOnEnter`/`triggerOnEnterSrv`** — a pre-existing gap that predates trap stadiums (on-enter abilities like intimidate/trace were *also* silently skipped for forced switches, just less noticeable before there was a stadium mechanic that visibly depends on it) but only surfaced once the trap stadiums made the omission observable, reported as "劇毒領域好像對新上的寶可夢沒起到作用". Fixed by adding `triggerOnEnter(opDeck[newIdx], opSide)` / `triggerOnEnterSrv(opDeck[newIdx], opRole, G, log)` right after the forced switch resolves — a forced switch is exactly as much a "field entry" as a voluntary one, it just wasn't routed through the same function. **When adding a new switch-adjacent mechanic, grep for every `G.playerIdx =`/`G.cpuIdx =`/`G[role+'Idx'] =` assignment site and check each one calls `triggerOnEnter`, not just the "obvious" ones (normal switch, KO-replacement)** — this is now the 2nd time a switch path silently skipped it.

**屬性轉換 (type-orb) is deliberately listed 6× in `TRAINERS`, not once** — consolidating 13 single-type orb cards into this one card (2026-07-06 pass) meant its raw per-slot draw probability dropped from "13 cards out of the pool" to "1 card out of the pool" — roughly a 10x reduction in how often the player would see *any* type-changing tool, even though each individual draw is now strictly more useful (no longer wasted on a mismatched type). Reported by the user as "一直抽不到屬性轉換" almost immediately after the consolidation shipped. Fixed by duplicating the plain object literal 6 times in the `TRAINERS` array. This is a general trap worth remembering: **consolidating N interchangeable single-purpose cards into 1 flexible card changes its draw frequency by roughly 1/N unless you explicitly compensate for it** — the fix for "make each draw better" and the fix for "keep the same draw frequency" are two different changes, and only doing the first one is a regression on its own.

**Weighted draw system (`weight` field, added 2026-07-20)** — the second real need for weighting arrived once the card pool grew to 68 type-restricted item cards across 18 types: the user wanted type-matching item cards to draw more often than generic item cards (屬性變換/傷藥/能量補丁 etc.), and plain duplication at that scale would have bloated `TRAINERS` by hundreds of entries. Added a proper `weight` field instead: every card object may carry an optional `weight` (default `1` if absent), and every consumer of `getDrawPool()`'s result that used to do `pool[Math.floor(Math.random()*pool.length)]` now calls a shared `weightedPick(pool)` helper (`pokemon_battle.html`/`server.js` each have their own copy, right after `getDrawPool()`) that sums `c.weight ?? 1` across the pool and picks proportionally — same pattern as the fishing feature's `rollFish()`. All 68 type-restricted **item** cards (`cat:'item'` with a `type` field) got `weight:10`; generic items and all stadium cards (even the few that carry a `type` field, e.g. `stadium-evil-forest`) were deliberately left at the implicit default of `1` — the user's comparison examples were all item cards, so the weighting was scoped to item-vs-item only. **`weight:10`, not a smaller number, was necessary because of the denominator**: a single-type Pokémon's draw pool has ~30 generic item entries at weight 1 (total weight 30) against only 4-5 type-matching item cards per type — at `weight:3` (the first value tried), even 5 cards × 3 = 15 was still far below 30, so type cards were actually *less* likely than generic ones in aggregate (verified empirically via a 20,000-sample `javascript_tool` distribution test — this is exactly the kind of assumption that needs the numeric check, not just "the individual card feels rarer so weighting must be working"). `weight:10` clears this comfortably (4×10=40 > 30) across every type, verified the same way. **When weighting a whole category of cards against a fixed pool of unweighted ones, calculate the required weight from the actual pool composition (count × weight ≥ opposing total) — don't guess a small multiplier and assume it's enough.** `type-orb`'s existing ×6 duplication was left untouched (different problem — compensating for lost absolute draw count after a 13→1 consolidation, not making a category "feel more common" relative to another category) and stacks fine alongside the new `weight` mechanism.

### Card system strategic-depth pass (2026-07-06)

User feedback: type-orb cards (originally 13 separate single-type `orb-*` items) were "在棄牌" (getting discarded unused) rather than played, because (a) the specific orb drawn rarely matched the current matchup's actual weakness, and (b) switching your attack's type away from your species' real type lost STAB, so the net gain after countering a resistance was often marginal — switching Pokémon was usually more decisive. Fixed by consolidating into **one flexible `type-orb` card** ("屬性轉換") that lets the player choose the type at play-time via a picker modal, **and** by making the overridden type automatically count as the attacker's own type for STAB purposes — `isOwnType = aBuff.typeOverride ? true : (...)` in `doAttack`, both engines. No downside anymore, only upside; CPU AI picks whichever type is super-effective (`bestMult > 1`) against the current opponent, or skips the card if nothing qualifies (kept deliberately simple, matching the project's established "CPU AI stays simple" precedent).

**Type-orb needs a type-selection UI** (`#type-orb-modal`/`#type-orb-grid`, one button per `Object.keys(TYPE_ZH)` — 18 types as of 2026-07-06, not the 16 the top-of-file project overview still documents from an earlier point) — intercepted in `useTrainer()`/PvP's `useTrainer()` the same way 瘋狂博士 is (don't consume the hand card until a type is actually picked). PvP sends `{type:'use_trainer', handIdx, chosenType}`; the server validates `chosenType` against `Object.keys(EFF)` **before** splicing the card out of hand (same "validate before consuming" ordering already used for 瘋狂博士's target validation — get this backwards and a rejected pick still eats the card).

**6 new item cards**, all reuse the existing `applyTrainer` switch (no new special-cased flows needed):
- **手牌破壞 (hand-wreck)**: opponent discards one random hand card; no-ops with a log message if their hand is empty (doesn't consume nothing — the card itself is still spent).
- **能量剝奪 (energy-drain)**: opponent loses 6 energy, floored at 0.
- **一擲千金 (gamble)**: 30% chance next attack ×4 (`Math.random() < 0.3`), 70% chance self-damage of 40% max HP, floored at 1 HP (matches `energize`'s existing self-damage-floor convention — cards never outright KO their own user).
- **背水一戰 (desperate-boost)**: `atkBonus += Math.round(50 * (1 - cur/hp))` — linear scaling, +0 at full HP up to +50 near death, unlike the flat-threshold `stadium-reversal` (+30 flat below 50% HP) it sits alongside.
- **連擊 (double-strike)**: `atkMult = max(atkMult, 1.4)` **plus** a `buff.doubleStrike` flag that makes the attack's own `atk.status` chance get rolled **twice** (independent rolls) instead of once — this is the one new mechanic that isn't just a number tweak. Implemented by capturing the flag as a local (`wasDoubleStrike` client-side; server-side the buff reset happens *after* the status roll in that file's code order, so `aBuff.doubleStrike` can just be read directly at roll time) and calling the status-roll logic a second time when set. Verified via 200-trial Monte Carlo against the extracted server code: a 50%-chance status rolled twice landed at a 0.75 proc rate, matching the `1 - 0.5²` independent-rolls prediction exactly — the client uses the exact same math (wraps the single-player's shared `tryInflictStatus()` call in an `if (wasDoubleStrike)` second call; server.js has no equivalent shared function, so the status-roll block there was pulled into a local `rollStatus()` closure specifically so it could be invoked twice).
- All 6 are `cat:'item'` (repeatable, drawn every turn) rather than `cat:'supporter'` — deliberate: unlike 搏命/瘋狂博士/全力出擊 (which are one-shot, hand-only tools), these are meant to be routinely-available tactical options, consistent with how `energize`/`reflect`/`switcher` (comparable power/complexity) are already items not supporters.

**2 new "trap" stadium cards** — this game's first entry-hazard-style mechanic (real Pokémon's Spikes/Toxic Spikes, reinterpreted for a persistent stadium instead of a per-side placed hazard): **尖峰陷阱 (`stadium-spikes`)** deals 15% max HP (floored at 1, never KOs) to any Pokémon that enters the field; **劇毒領域 (`stadium-toxic-field`)** poisons any Pokémon that enters the field (skipped if it already has a status). Both apply **symmetrically to both sides** (same "global rule, not tied to whoever played the card" convention as every other stadium in this game) and are gated behind a **new `isFieldEntry` parameter on `triggerOnEnter`/`triggerOnEnterSrv`** (defaults to `true`): every genuine bench→field entry point (initial battle start, normal/free switch, KO-replacement on both sides) already called this function to fire on-enter abilities like intimidate/trace, so the trap logic (`triggerTrapStadium`/`triggerTrapStadiumSrv`) piggybacks on the exact same call sites for free — **except** 瘋狂博士's in-place transform and Mega Evolution, which mutate the *same* battle-slot Pokémon without it ever leaving the field, and now explicitly pass `isFieldEntry=false` at their `triggerOnEnter(...)` call sites so a trap stadium doesn't punish playing those cards. If you add a genuinely new "enters the field" call site in the future, it defaults to triggering traps automatically; if you add a new *transform-in-place* mechanic, remember to pass `false` explicitly or it'll incorrectly eat trap damage/status on activation.

**Second follow-up batch, same day**: user reported the fix above was found to have 2 more gaps (see the "attacker dies without landing a hit" and "缺少一份既有清單的其中一處" lessons below — same class of mistake, new instances), then asked for a further round of expansion:
- **羅馬鬥技場 (`stadium-colosseum`)**: fighting-type moves deal ×1.5 (a new `colosseumMult` factored into the final `dmg`/`damage` formula in `doAttack`, both engines, alongside `megaBoostMult`), **and** fighting-type attacks are no longer fully immune against ghost-type (`EFF.fighting.ghost` is normally `0` — `effActive`/`srvEffActive` now override that specific pairing back to `1`/neutral when this stadium is active, added right after the existing 邪惡森林/`stadium-evil-forest` override in the same function, following its exact pattern).
- **掠奪 (plunder)**: steals one random card from the opponent's hand **into your own** — different from 手牌破壞/hand-wreck, which just discards the opponent's card into nothing. Since this *adds* to the plunderer's hand, it can push them over the 5-card cap; single-player calls the existing `showDiscardModal(() => {})` if `G.playerHand.length > 5` after stealing (no-op callback since nothing needs to happen once they've discarded); PvP sets the existing `G[role+'NeedsDiscard']` flag (same field `drawForRole` already sets after a normal draw) rather than needing a new mechanism.
- **通訊封印 (comm-seal)**: opponent can't play `cat:'supporter'` cards during their *immediately next* turn. Implemented as a "promote-then-consume" pair of flags rather than trying to align with a turn-end hook: playing the card sets `G[opSide+'SupporterLocked'] = true` (persists indefinitely until consumed); the **start of that side's own next turn** (single-player: top of `drawAfterTurn`; PvP: top of `drawForRole`, which every genuine turn-start path already funnels through) promotes it to a separate `...LockedThisTurn` flag and immediately clears the original — so the lock is "live" for exactly the turn it was promoted into, and automatically off for every turn after. `useTrainer()`/PvP's `use_trainer` handler check the `...ThisTurn` flag (not the raw one) when gating supporter-card play; single-player's CPU equivalent (`cpuUseTrainers()`) reads-and-clears the raw flag directly at its own top, since CPU's trainer-card usage is one batch call per turn rather than per-click like the player's, so it doesn't need the two-flag promotion dance — read the flag once, decide, done. If you add a third "opponent skipped one of my turns" effect, decide up front whether it's more like a *batch-per-turn* consumer (CPU-AI style: read-and-clear once) or a *scattered-across-the-turn* consumer (player-UI style: needs the promote-at-turn-start pattern) — using the wrong one either double-clears too early or never clears at all.

**Two more instances of the "added a new entry to one enumerated list but not its sibling list" bug class** found while implementing the batch above (see pokemon-data skill's moveset-diversification note for the general lesson): the type-picker modal's CSS needed *both* a `display:none` base rule *and* a `.open { display:flex }` rule added to their respective shared selector lists — single-player's copy only got the first one added when `#type-orb-modal` was introduced, so the modal would open (`classList.add('open')`) but stay visually `display:none` forever; reported as "屬性轉換沒辦法點". Same root cause as the stadium-activation `switch` case gap documented in the "Card system strategic-depth pass" section above (adding a new `cat:'stadium'` entry to `TRAINERS` without also adding its id to the enumerated `case 'stadium-x': case 'stadium-y': ...` list in `applyTrainer`) — **whenever a new card/element needs to slot into an existing multi-selector CSS rule or an existing `case A: case B: case C:` list, grep every existing member of that list, not just the one that looks most relevant, and add the new one to *all* the sibling locations in the same edit.**

**Third stadium batch, same day — 4 more cards, no user-reported bugs this round** (the accumulated "grep every sibling list" discipline from the two prior batches paid off — each new stadium id was verified present in all 4 relevant spots — `TRAINERS` entry, the `case 'stadium-x': ...` activation list, `effActive`/`srvEffActive`, and the `doAttack` multiplier chain — in both files before testing, catching zero omissions this time instead of finding one after the fact):
- **魔幻空間 (`stadium-mystic-space`)**: psychic-type defenders take ×0.9 damage (new `mysticSpaceMult`, defender-type-based like `thickFatMult`/`solidRockMult`, not attacker/move-type-based) **and** lose their weaknesses — `effActive`/`srvEffActive` override any `m > 1` back to `1` when the defender is psychic, following the exact `stadium-evil-forest` pattern (grass-type immune to disadvantages) rather than the colosseum pattern (which only fixes one specific immunity pairing).
- **熔岩火山 (`stadium-lava`)**: fire-type moves ×1.1, water-type moves ×0.5 — both are **move-type-based** (`atkType`), not the attacking Pokémon's own species type, matching how `stadium-colosseum`'s fighting boost was implemented (established precedent, kept consistent rather than switching interpretations card-to-card even though the user's Chinese phrasing for different cards inconsistently says "X系寶可夢" vs "X屬性的技能" — read literally those would imply different bases, but this game already established the "move-type" reading for this class of effect and switching per-card would create inconsistent, surprising interactions).
- **海洋世界 (`stadium-ocean`)**: water-type moves cost half energy — **new stadium-based cost reduction**, no prior precedent existed (only `pressure`'s cost *increase* did). Added to `effectiveCost`/`effectiveCostSrv`, which single-player already threaded through every affordability check and deduction site, so no other call sites needed touching — but **PvP's `effectiveCostSrv` didn't take `G` as a parameter at all before this** (it only needed `opponentPoke` for the `pressure` check), so this required adding a third parameter and updating its one call site in the `attack` handler; if you extend a helper's signature like this, grep all call sites — there was only one here, but a helper with several callers would silently break at every uncalled-through site otherwise. Electric-type moves also ×1.5 damage (`oceanMult`, same move-type-based pattern as lava's fire boost).
- **莊嚴神社 (`stadium-shrine`)**: normal-type moves are always treated as maximally super-effective (`m = 2`), an unconditional override rather than a modifier — added as the last check in `effActive`/`srvEffActive` (after the colosseum/mystic-space adjustments) so it wins regardless of what any other stadium or the base type chart would have said, including forcing damage through normal-type's real immunity against ghost (`EFF.normal.ghost === 0` normally).

Verified via the same two-track approach used throughout this session: single-player live browser attacks with hand-picked matchups (verified exact expected damage against the full multiplier chain including STAB and any relevant defender ability like thick-fat, not just the new stadium term in isolation — e.g. 魔幻空間 test used 妙蛙花→胡地 for the weakness-removal check and confirmed `48×0.9=43` exactly; 熔岩火山 test incidentally exposed that 卡比獸/Snorlax has `thick-fat`, which the expected-value math had to account for, `35×1.5×0.6×1.1=34`); PvP verified by extracting `srvEffActive`/`effectiveCostSrv`/`doAttack` into a standalone Node module and asserting exact damage/cost outputs for all 4 new stadiums, matching the single-player browser results number-for-number.

**Bug found and fixed 2026-07-13**: `public/pvp.html` has its *own* client-side `effectiveCost(atk, opponentPoke)` (used purely for the attack-button display number and the local afford-check that grays out a button) — separate from `effectiveCostSrv` in `server.js`, which is what actually validates/charges the energy. When 海洋世界 was added to `effectiveCost`/`effectiveCostSrv` (the single-player function and the server function), **`pvp.html`'s client-display copy was missed** — it only ever had the `pressure` +2 term, never the ocean ×0.5 term. Net effect: during PvP with Ocean World active, the button showed/enforced the *full* (un-halved) cost while the server silently charged the *correct* (halved) cost — a display/reality mismatch, not a scoring bug (the user's phrasing was "should be the *required* energy that's halved, not just the energy actually spent" — i.e. the discount was real but invisible/not reflected in what the UI told the player they needed). **Lesson: `effectiveCost`/`effectiveCostSrv` is not a two-copy pair, it's a three-copy pair** (single-player, server, PvP-client-display) — this skill's own energy-system section above already listed "pvp.html's renderActions()/playerAttack() client-side mirrors" as call sites needing the helper, but didn't flag that the *helper itself* is duplicated a third time and must be kept in sync manually (no shared module to import from, this is a single-file-per-engine codebase). When adding a future stadium/ability that changes move cost, grep for `function effectiveCost` (should get exactly 3 hits: `pokemon_battle.html`/`public/single.html` share one via the sync check, `server.js`, and `public/pvp.html`) and update all three, not just the two you'd naturally think of (single-player + server).

## Status effects

Shape: `{type, turnsLeft}`. `turnsLeft`: sleep 2-3, confusion 2-4, freeze 2 (all random/fixed); poison/burn/paralysis use `999` (i.e. "persists until cured", not turn-limited — paralysis has no countdown at all, just a flat 50%-skip roll every attack attempt regardless of `turnsLeft`).

**Two-phase model (reworked 2026-07-03)** — status resolution is split into "does it block acting" (only relevant when *attempting to attack*) and "does it tick/deal damage at turn end" (relevant on every turn regardless of what the Pokémon did):

- **Attack-gating** (`handleAttackerStatus()` single-player / `handleStatus()` PvP, called only from `playerAttack`/`cpuAttack`/the `attack` WS handler): sleep/freeze/paralysis may block the attempt (`onSkip`); confusion may force a self-hit instead of the intended move. Poison/burn **never** resolve here anymore — they always let the attempt through untouched.
- **End-of-turn damage** (`applyEndOfTurnStatus()` single-player / `applyEndOfTurnStatusSrv()` PvP): applies poison/burn damage (or poison-heal ability healing) *after* the turn's action has already resolved — whether that action was a landed attack, an attack blocked by sleep/paralysis/freeze, or a standby. This is why a poisoned Pokémon at 1 HP still gets to swing before it faints, instead of dying before its move — matches mainline Pokémon timing, was a deliberate 2026-07-03 fix per user report ("中毒的寶可夢還沒攻擊就倒下了").
- **Non-attack duration tick** (`tickNonAttackStatus()` single-player / `tickNonAttackStatusSrv()` PvP): decrements sleep/freeze/confusion `turnsLeft` on a turn where the Pokémon didn't attempt to attack (standby, or single-player's CPU-can't-afford-any-move branch) — no blocking, no confusion self-hit, just the countdown (and wake-up message) so standby genuinely progresses toward recovery instead of leaving the counter frozen forever.

Every turn-ending call site needs to call the right combination of these — grep existing call sites in `playerAttack`/`cpuAttack`/`playerStandby` (single-player) and the `attack`/`standby` WS handlers (PvP) before adding a new one. The trickiest wrinkle: applying poison/burn *after* an attack can newly kill an attacker that survived the attack exchange itself — in PvP this is handled by applying the tick and *then* computing `attackerDied`, so the existing double-KO/attacker-only/defender-only/neither branching (see KO resolution below) picks it up automatically; single-player does the equivalent via a shared `tickAndHandoff`/`tickAndEndTurn` closure that wraps whichever path the turn took.

CPU will self-cure **any** bad status (including freeze/sleep) with 萬能藥/治療師 if it has one in hand, before even attempting to attack — this is intentional pre-existing AI behavior, not a bug, even though it can look like "the status did nothing." Confirmed with the user this should stay as-is (2026-07-02).

## Damage calc & buffs — `doAttack()`

Buff object `{atkBonus, atkMult, shield, typeOverride, reflect}`, one per side (`G.playerBuff`/`G.cpuBuff` or `G[role+'Buff']`). All four of atkBonus/atkMult/typeOverride/shield reset the instant an attack actually happens (consumed regardless of hit/miss). `atkMult` used to only ever go *up* (trainer-card boosts, via `Math.max(buff.atkMult, N)`) until `intimidate` started pushing it *down* (`Math.min(opBuff.atkMult, 0.5)`, 2026-07-04) — same "resolve to the strongest applicable value in that direction" pattern, just mirrored for a debuff; a value below 1 in the UI's buff-tag badge now needs its own visual treatment (`.buff-tag.debuff`, red) distinct from the existing boost styling (`.buff-tag.mult`, orange) so a player can tell at a glance whether their next hit is buffed or debuffed. `reflect` is different — it's meant to persist until the *next* attack lands on that side, however many turns that takes, EXCEPT it must expire if the opponent's turn passes without attacking (switch, standby, or a status-skip) — every one of those turn-ending paths needs an explicit `xBuff.reflect = false`; there is no central place this happens automatically, so if you add a new way for a turn to end without an attack, check whether reflect needs clearing there too (grep `reflect expires` for the existing sites, single-player and server.js each have ~4).

Ability hooks are threaded through `doAttack` two ways — pick whichever fits the effect (see pokemon-data skill for the full roster of which ability uses which):
- **Post-hit** (`triggerAttackerAbility`/`triggerDefenderAbility` single-player, `*Srv` PvP): fired *after* damage is already applied, for effects that don't change the damage number itself (静電/static, 毒刺/poison-point inflicting a status; 粗糙皮膚/rough-skin dealing separate recoil). Simplest, least risky to add.
- **In-formula** (added 2026-07-03 for 9 new abilities): `abilityDmgMult` (attacker-side: 堅韌/guts, 大力士/huge-power, 激流-family/blaze-boost), `stabMult` (適應力/adaptability bumps it from 1.5→2), and a `defAbilityMult` product (厚脂肪/thick-fat, 硬岩/solid-rock, 神秘之守/frisk-ward) are all computed *before* the `dmg =` line and multiplied directly into the formula. Two abilities don't fit even that: `water-absorb` (儲水) is an early-return branch right after the `reflect` check — full immunity, heals instead, never reaches the damage formula at all — and `sturdy` (頑強) captures `wasFullHp = defender.cur === defender.hp` *before* the formula runs, then clamps `defender.cur` to 1 right after it's decremented, but only if that pre-hit snapshot was true. If you add another ability that needs to know pre-hit state, follow the `wasFullHp` pattern — capture the snapshot before any mutation, not after.

`triggerOnEnter`/`triggerOnEnterSrv` fire wherever a Pokémon becomes newly active (battle start, switch, KO-switch) — grep call sites before assuming you've covered every "a Pokémon just entered battle" moment, there are ~4-5 distinct call sites per engine. `intimidate` lives here: sets `opBuff.atkMult = Math.min(opBuff.atkMult, 0.5)` (changed 2026-07-04 from a flat `atkBonus -= 15`, deemed too weak — note the `Math.min` rather than straight multiplication, so it can't stack below ×0.5 or interact multiplicatively with an existing boost). It used to also check whether the opponent's active had `clear-body` before applying the debuff — that ability-vs-ability interaction pattern was deleted along with `clear-body`'s retirement the same day; if a future ability needs to veto another Pokémon's onEnter effect, the pattern (check the *opponent's* current active's ability inside the triggering ability's own code, not a passive block registered on the other side) still applies, just nothing currently uses it.

**Stale `ability.desc` text, fixed 2026-07-13**: the 2026-07-04 `atkBonus -= 15` → `×0.5` logic change updated the *behavior* everywhere, but only some of the 8 intimidate-carrying Pokémon's data entries got their `ability.desc` string updated to match at the time (大嘴娃/雷電獸/暴飛龍/頭巾混混 said "×0.5"; 暴鯉龍/穿山王/風速狗/布魯皇/肯泰羅/阿柏怪/流氓鱷/姆克鷹 still said the old "威力 −15"). The logic was never wrong — this was purely a cosmetic desc-string drift across all 3 engine files (`pokemon_battle.html`, `public/single.html`, `server.js`), each Pokémon's `ability.desc` is a hand-written literal, not derived from the trigger code, so a behavior change doesn't propagate to it automatically. **Lesson: when changing an ability's *effect*, grep every `ability:{id:'that-id', ...}` data-entry occurrence across all 3 files and check `desc` text matches the new behavior** — don't assume updating the one `triggerOnEnter`-adjacent line that computes the effect is sufficient, the flavor-text copy is a separate, easy-to-miss source of truth that has to be kept in sync by hand.

- **Status-infliction-time** (added 2026-07-04): `tryInflictStatus()` single-player / the inline status block inside `doAttack` in server.js — this is where a defender's ability can veto an incoming status (`own-tempo` blocks confusion, `insomnia` blocks sleep — same pattern, just a different `effect` string check) or piggyback on a successful one (`sync-status` copies poison/paralysis/burn onto the attacker after it lands on the defender). Any *card* that inflicts a status by directly setting `.status = {...}` instead of going through this function (混亂藥/confuse-potion, 瓦斯攻擊/gas-attack, 火焰彈/fire-bomb, 絕對零度/absolute-zero) bypasses these checks entirely — if a new status-blocking ability needs to also cover a specific card, add the same guard to that card's `applyTrainer` case individually (already done for `own-tempo` + confuse-potion, not yet done for the others — no Pokémon needs it yet).
- **Type-immunity-with-a-payoff** (the `water-absorb` early-return pattern, extended 2026-07-04): `motor-drive` (immune to electric, +3 energy to the defender's side instead of a heal) and `flash-fire` (immune to fire, `dBuff.atkBonus += 20` instead of a heal) both copy `water-absorb`'s early-return-right-after-the-`reflect`-check shape but swap out the payoff. If you add another type-immunity ability, this is the template: early return, no damage, `return`/`return { damage: 0, mult: 1 }` before the normal formula runs, and don't forget to still reset `aBuff`'s consumed fields (server.js does; single-player's older `water-absorb` block doesn't bother — a pre-existing minor inconsistency between engines, not something introduced 2026-07-04, low-risk since it only matters if the attacker had a leftover buff queued right before hitting an immune target).

Multiple Pokémon commonly share one `ability.id` **and can have different `name`/`desc` text per entry** — the dispatch is by id (`if (attacker.ability?.id === 'guts')`), not hardcoded per species or display name, so giving two different Pokémon the same ability (even under different in-game names, e.g. Overgrow/Torrent/Blaze all share `blaze-boost`) is the normal case, not a special one. When a real ability doesn't map to any existing effect and isn't worth a bespoke new one, reusing the closest existing id with a relabeled `name`/`desc` (see pokemon-data skill's ability table for examples like 超幸運→`huge-power`, 隔音→`frisk-ward`) is an accepted, deliberate shortcut — not a bug.

## Energy system (2026-07-02)

Every side has energy (`G.playerEnergy`/`G.cpuEnergy` single-player, `G.p1Energy`/`G.p2Energy` PvP), 0-20, starts at 5, `+3` capped at 20 at the start of *that side's own turn* — single-player hooks this into the exact same two spots that already fire once per turn-start (`cpuAttack()`'s top for CPU, `drawAfterTurn()` for the player); PvP hooks it into `drawForRole(G, role)`, which is already the single canonical "this role's turn is genuinely starting" function (correctly handles the KO-switch-queue-deferred case too — see pvp-networking skill). Resets to 5 on a new single-player stage / new PvP game, same place HP gets fully healed.

Each move's `cost` (see pokemon-data skill) is deducted right before `doAttack` actually fires — never on a status-skip. **PvP validates energy server-side** in the `attack` WS handler (`if (G[role+'Energy'] < atkCost) reject`) before trusting `msg.idx` — client-side button disabling is a UX nicety only, not the real guard. CPU (single-player only) filters its move choices down to `cost <= G.cpuEnergy` before applying its usual damage-preference logic; if literally nothing is affordable it skips the attack via the same `onSkip`/`endTurn` path status-skips already use. In practice this branch rarely fires — every tier's weak-move cost ceiling is ≤ the flat +3 regen, so some move is affordable right after any turn-start regen by construction.

**`effectiveCost(atk, opponentPoke)` single-player / `effectiveCostSrv(atk, opponentPoke, G)` server.js / `pvp.html`'s own client-display copy** is the one place a move's *effective* cost can differ from its base `cost` — currently only the 海洋世界 stadium's water-cost-halving. **Every place that checks or deducts a move's cost must go through this helper, not `atk.cost` directly** — `playerAttack`/`cpuAttack`'s affordability checks and deductions, `renderActions()`'s button-disable + displayed number, PvP's `attack` WS handler (validation *and* deduction — both use the same computed `atkCost` local so they can't drift apart), and `pvp.html`'s own `effectiveCost`/`renderActions()`/`playerAttack()`. If a future ability or card needs to affect energy costs, extend this helper rather than duplicating the tax logic at each call site.

**壓迫感/`pressure`'s continuous +2 tax — added 2026-07-04, removed 2026-07-14, not just disabled.** For 10 days this helper also added `Math.min(20, atk.cost + 2)` whenever the opponent's active had `pressure` (on top of whatever `pressure`'s own onEnter-drain already did — see pokemon-data skill). The user asked for the continuous tax removed while keeping the onEnter drain (a mix-up with `intimidate` while reviewing the ability's real effect surfaced this) — all 3 copies of the helper had the `opponentPoke?.ability?.id === 'pressure'` branch deleted outright (not commented out, not gated behind a flag), so `pressure` is back to being a pure onEnter-only ability. If similar "opponent's move costs more while I'm the active defender" effects come up again, this needs new code, not an old switch to flip back on.

Energy-granting cards: 3 items (`energy-patch-s/m/l`, +2/+3/+4) and 1 supporter (`cheerleader`, fills to 20) — same `applyTrainer` switch as everything else. The discard-2-for-1 trade modal (`chooseTrade()`/PvP `discard_trade`) has a third option beyond stadium/item: `cardType:'energy'` grants +5 instead of drawing a card.

## Mega Evolution (2026-07-04, energy model reworked same day)

**The Mega gauge is a per-side shared resource, not per-Pokémon** (reworked from an initial per-instance design the same day the feature shipped, per explicit user request: "讓mega 條跟能量條一樣，所有寶可夢共用"). Lives on `G` exactly like move-energy: `G.playerMegaEnergy`/`G.cpuMegaEnergy` (0-20) single-player, `G.p1MegaEnergy`/`G.p2MegaEnergy` PvP, plus a one-time-use gate `G.playerMegaUsed`/`G.cpuMegaUsed` (`p1MegaUsed`/`p2MegaUsed` PvP). The battle-instance clone itself only keeps `poke.megaEvolved` (bool, `undefined` if the species has no `mega` data) — there is no more per-Pokémon `megaEnergy` field. Only 23/100 Pokémon have `mega` data (see pokemon-data skill); everything below no-ops for the rest.

**One mega evolution per side per battle, full stop.** Once `G.xMegaUsed` is set, the gauge is gone for the rest of that battle even if the team has other Mega-capable Pokémon on the bench and even if the gauge could theoretically refill — there is no per-Pokémon accounting to get around this, `xMegaUsed` is the single source of truth checked everywhere (accumulation, the evolve button, the WS handler, the UI's show/hide).

**Accumulation** — inside `doAttack`, right after damage is computed and applied (both engines): `megaGain = Math.max(1, Math.round(damage / 25))`, added to **only the attacker's** gauge, gated by `!G[aSide+'MegaUsed']` — **not** by whether the attacking Pokémon currently on the field has `.mega`. The gauge charges from combat regardless of who's fighting, same as move-energy; a team of three non-Mega Pokémon technically still charges a gauge, it's just going to be moot since no active Pokémon can ever be eligible to spend it. Single-player determines the attacker's role via the `aSide` string already passed into `doAttack`; server.js has no such string, so it back-derives it by identity-comparing `aBuff` against `G.p1Buff` (same trick already used for `motor-drive`'s energy refund). **The defender does not gain anything from taking damage** — an earlier version of this feature credited both attacker and defender identically from every exchange, which the user found made the two sides' gauges look linked/synced in practice (they moved in near-perfect lockstep turn after turn, even though they were genuinely separate variables) — changed 2026-07-05 to attacker-only per explicit request, so each side's gauge now visibly reflects "damage *I* dealt," not "damage anyone dealt near me."

**Triggering** — `applyMegaEvolution(poke, side)` single-player / the inline `mega_evolve` WS handler in server.js: mutates `poke.id` (→ `mega.spriteId`), `poke.type`/`type2`, and `poke.ability` (→ a full copy of `mega.ability`), sets `megaEvolved = true`, **bumps `poke.hp` by ×1.2 (rounded) then fully heals `poke.cur = poke.hp` and clears `poke.status = null`**, sets the side's `MegaUsed` flag, then calls `triggerOnEnter(poke, side)` so 複製/`trace` can fire immediately, then kicks off the evolution animation (see below). **This is a free action — it must never end the turn**, same as before the rework. Gate conditions (duplicated in the UI's button-disable, same "client is a UX nicety, server is the real guard" pattern as move-energy): `poke.mega && !poke.megaEvolved && !G[side+'MegaUsed'] && G[side+'MegaEnergy'] >= 20`; PvP additionally requires `G.turn === role && !G.pendingKOSwitch`.

**Universal Mega power boost** (added 2026-07-05, after user feedback that Mega Evolution "didn't feel much stronger" — many of the reused abilities across the 87 Mega-capable species are situational/defensive rather than reliably impactful, e.g. `frisk-ward`'s 25% proc or `solid-rock`'s supereffective-only reduction, so a lot of evolutions changed little turn-to-turn). Two flat multipliers apply on top of whatever the species' specific Mega ability does, independent of `ability.id`: **+20% max HP** (the `poke.hp *= 1.2` bump above, applied once at evolve time — a real HP increase, not a shield or heal-only effect, so it also raises how much damage the Pokémon can soak afterward) and **+10% flat attack damage**, via `megaBoostMult = attacker.megaEvolved ? 1.1 : 1` multiplied into the final `dmg`/`damage` calculation in `doAttack` (both engines), stacked alongside `abilityDmgMult`/`defAbilityMult` rather than folded into either — keeps it simple to reason about ("Mega always adds +10% on top, no matter what the ability does") and means it doesn't need touching if a species' Mega ability is later redesigned. Deliberately *not* implemented as a per-species tunable, to keep all 87 Mega-capable Pokémon consistent — if a specific species still feels weak after this, prefer adjusting its Mega `ability`'s own effect over adding a per-species exception to this multiplier.

**Persistence** — a specific Pokémon's evolved state still sticks to that battle-instance object for the rest of the battle regardless of switching (unchanged from the original design) — but since the whole *team* only gets one use, at most one Pokémon per side will ever carry `megaEvolved: true` in a given battle.

**UI visibility** — the Mega bar shows per side whenever `!G[side+'MegaUsed'] && deck.some(p => p.mega)` (checks the *whole roster*, not just the active Pokémon) — `renderSide(side, poke, deck, activeIdx)` already receives `deck` for this. The evolve button additionally requires the *currently active* Pokémon to itself have `.mega` (you can only evolve who's currently out).

**Mega Prism Tower** (`stadium-mega-prism`, added same day): grants **+10 Mega energy to whichever side's own turn is starting**, hooked into the exact same two turn-start points the +3 move-energy regen already uses (single-player: `cpuAttack()`'s top for CPU, `drawAfterTurn()` for the player; PvP: `drawForRole(G, role)`) — gated on `!G[role+'MegaUsed']` so it stops mattering once a side has used its one evolution. Like all other stadiums in this game, it's a global rule both sides benefit from symmetrically, not tied to whoever played the card.

**Evolution animation — full-screen cutscene** (redesigned 2026-07-05, replacing an earlier in-card version; see the retired-approach note below for why): `playMegaEvolveAnim(side, poke)` single-player / `playMegaEvolveAnim(side)` PvP (PvP derives `poke` internally via `side === 'player' ? P() : CP()` — the state is already applied via `applyServerState()` by the time it runs). Both open `#mega-cutscene-modal`, a `position:fixed;inset:0` full-screen takeover (same show/hide-via-`.open`-class pattern as every other modal in this game, but at a higher `z-index:200` so it sits above everything including other modals) containing three elements inside `.mega-cs-stage`: `#mega-cs-stone` (a 💎 emoji that scales/rotates in, holds a pulsing glow, then "shatters" — scales up + flashes white + fades), `#mega-cs-sprite` (a dedicated `<img>`, **not** the battle card's `#player-sprite`/`#cpu-sprite` — src is set to `spriteUrl(poke.id)` only at the shatter moment, then pops in with `.reveal`), and `#mega-cs-title` (fades in after the sprite reveal, showing "`{name}` Mega 進化！"). Timeline (both engines identical): open+stone-appear at t=0, stone starts pulsing at 600ms, shatter+16-particle-burst+sprite-reveal at 1300ms, title at 2000ms, modal closes at 3200ms — after which the ordinary battle screen underneath (already fully updated, since `applyMegaEvolution`/`applyServerState` ran before this was called) is simply what's left. No `G.busy` handling needed to block input during the cutscene — the fixed-position overlay already intercepts all clicks by sitting on top of the DOM.

**Why not animate the in-battle sprite directly (retired approach)**: the first version wrapped/burst *on* the `.b-card` element using the actual `#player-sprite`/`#cpu-sprite` `<img>`, and reused that image for both the small in-card effect and the underlying idle/breathing animation. That coupling caused a real bug (`el.style.animation` inline styles silently suppressing `setSprite()`'s own `.sprite-enter` class-animation, whose `animationend` then never fired, permanently blocking `syncSpriteAnim()` from ever re-adding `sprite-idle` — reported by the user as "mega 進化的寶可夢不會動"). Using a **separate, dedicated `<img>` inside the cutscene modal** sidesteps this category of bug entirely — the cutscene never touches `#player-sprite`/`#cpu-sprite`'s classes, so it can't collide with `setSprite`/`syncSpriteAnim`'s lifecycle no matter how its own animation is implemented. If you build another full-screen or otherwise-isolated Pokémon-image effect in the future, prefer a dedicated image element over reusing the battle card's own sprite — it removes a whole class of animation-state bugs, not just the inline-style one.

**CPU AI** (single-player only, no PvP bot): `cpuAttack()`'s very first lines check `CP().mega && !CP().megaEvolved && !G.cpuMegaUsed && G.cpuMegaEnergy >= 20` and auto-apply before any move-selection logic runs — greedy/immediate, no timing strategy, matching this project's established "CPU AI stays simple" precedent (see 搏命/瘋狂博士 in the Trainer cards section above).

**4 new ability mechanics were needed** beyond simply flagging a mega-form's real ability (most Mega abilities reuse an existing effect id with a relabeled `name`, exactly like the legendary-weather-ability pattern — see pokemon-data skill's Abilities section for the full per-species table):
- `tough-claws`: flat `abilityDmgMult × 1.3` on every move (real Tough Claws only boosts contact moves; this game has no contact/non-contact move tagging, so it's simplified to apply universally).
- `technician`: `abilityDmgMult × 1.5` only when `atk.dmg <= 60` — maps cleanly onto this game's existing weak/strong move-power tiers (weak moves are always ≤48, strong always ≥65), so the real "boosts moves ≤60 power" rule needed zero reinterpretation.
- `mold-breaker`: attacker-side flag (`const moldBreaker = attacker.ability?.id === 'mold-breaker'`) that forces `thickFatMult`/`solidRockMult`/`friskWardMult`/`multiscaleMult` to `1` and skips the `sturdy` 1-HP save, regardless of the defender's actual ability — implemented by threading `!moldBreaker &&` into each of those four conditions plus the sturdy check, rather than a separate code path.
- `trace`: lives inside `triggerOnEnter`/`triggerOnEnterSrv` (not `doAttack`) — copies the opponent's *current* active ability (`{...opPoke.ability}`) onto self. Fires both on a normal switch-in and (via the explicit `triggerOnEnter` call inside `applyMegaEvolution`) at the instant of evolving, so Alakazam's Mega form immediately traces whatever the opponent has out at that moment rather than waiting for a future re-entry.

## KO / double-KO resolution — the trickiest part, read before touching

A Pokémon's death can come from three independent sources in the same exchange: (1) the direct attack damage to the defender, (2) reflect bouncing damage back onto the attacker instead, (3) a defend-ability's recoil (currently only 粗糙皮膚/rough-skin) hitting the attacker *in addition to* the defender taking normal damage. (1)+(2) can't both happen (reflect fully redirects, defender takes zero direct damage that exchange). (1)+(3) **can** both happen — a genuine simultaneous double-KO.

Both engines had (separately discovered, separately fixed 2026-07-02) bugs where checking only the attacker's death and returning early would silently skip ever checking whether the defender *also* died in the same exchange — PvP's version was worse, since it could leave the game state stuck (no winner declared, no forced switch) rather than just misattributing a winner. The fix pattern, if you touch this again: compute **both** `attackerDied`/`defenderDied` booleans up front, handle all 4 combinations explicitly (neither / attacker-only / defender-only / both), and for the both-died case check each side's *remaining team* alive-count independently before deciding win/loss/draw — don't just reuse the single-sided `handleKO`/`pendingKOSwitch` early-return logic, it assumes only one side could possibly be wiped. Single-player has no draw concept — any true double-team-wipe resolves as a player loss (`showResult('cpu')`), matching the precedent set by 搏命. PvP does support `G.winner = 'draw'`.

`bothTeamsWiped()` (single-player helper) and the inline `roleAlive`/`opAlive` checks (PvP) are the canonical way to ask "did both sides just run out of Pokémon" — reuse them rather than re-deriving.

**PvP-only bug, fixed 2026-07-08: `G.turn` was never transferred to the opponent when the attacker died without the defender dying** (reflect bouncing lethal damage back onto the attacker, or the attacker dying from its own confusion self-hit before ever landing a hit — `sResult.died`). Both branches correctly set `G.pendingKOSwitch = role` so the attacker picks a replacement, but neither set `G.turn = op` the way the sibling `defenderDied`/both-died branches right next to them already did — so once the attacker resolved their `ko_switch`, `G.turn` was still `role`, and `ko_switch`'s own `if (G.turn === role) drawForRole(G, role)` check let the *same* player draw and act again immediately, reported by the user as "反彈致死後應該換對方回合". Fix: both branches now also set `G.turn = op; G.round++; drawForRole(G, op);` right there (the opponent's own Pokémon is fine, so there's no reason to wait for a `ko_switch` round-trip before giving them their turn — contrast with the `defenderDied` branch, which correctly *does* wait, because in that case it's the opponent's own Pokémon that needs replacing first). The `sResult.died` branch additionally expires the opponent's `reflect` buff (`G[op+'Buff'].reflect = false`) — same "reflect expires if the attacker's turn ends without them actually attacking" rule already applied to the `sResult.skipped` branch just above it, since dying to your own confusion counts as never attacking. **Single-player never had this bug** — it chains callbacks synchronously (`handleKO('player', () => setTimeout(cpuAttack, 700))`) rather than checking a `G.turn` variable, so control flow can't get stuck on the wrong side the way a stateful turn-flag can. If you find a *third* "attacker dies without the defender dying" branch in the future, check it sets `G.turn = op` — this was the second time the same omission was found in sibling branches.

## Moveset overhaul: cost-tier system + 8 support moves (2026-07-14)

All 153 Pokémon's movesets were regenerated from scratch by a one-shot Node script (not hand-edited — see pokemon-data skill for the HP→tier distribution table and generation approach), replacing the old "2 weak + 2 strong (+ occasional mid-tier)" moveset shape with a 4-cost-tier system, and one of each Pokémon's 4 move slots is always overwritten with one of 8 new **support moves** (0-damage, pure-effect). This is the largest single data rewrite in the project's history (612 move entries across 2 engines) and introduced several genuinely new move-level fields and mechanics that didn't exist before.

**New per-move data fields** (all optional, `undefined` = doesn't apply):
- `megaBoost: true` — tier1 (cost 0-3) moves only. Doubles Mega-energy gain from that hit (`megaGain = Math.max(1, Math.round(damage/25)); if (atk.megaBoost) megaGain *= 2;` — both engines, right where mega energy already accumulates in `doAttack`).
- `bonusEnergy: N` — tier2/3 (cost 4-15) moves. Using the move sets `G[role+'BonusEnergyNextTurn'] += N`, consumed (added on top of the normal +3 regen, then zeroed) in `drawAfterTurn`/`drawForRole` — the exact same promote-then-consume pattern as 通訊封印's `SupporterLocked`→`SupporterLockedThisTurn`.
- `selfHeal: 0.15-0.30` — tier4 (cost 16-20) moves, roughly half of them (the other half get a `status` instead — see below). Heals `round((attacker.hp - attacker.cur) * selfHeal)` — a percentage of the attacker's *missing* HP, not of damage dealt — applied right after status infliction in `doAttack`, gated on `damage > 0`.
- `support: true` + `effect: 'brace'|'sword-dance'|'thief'|'shadow-dance'|'debuff'|'meditate'|'trick'|'focus-energy'` — marks a move as one of the 8 support moves (see below). These have **no `dmg` field at all** (not `dmg:0` — genuinely absent) — every UI spot that displays `a.dmg` needed a `a.support ? '輔助'/'威力'+a.dmg` guard (`renderActions`/`renderSelCard`/`showBenchPopup`/Pokédex in both engines, `renderActions`/team-select/bench-popup in pvp.html) or it prints `undefined`.

**Move-generation approach** (script, not committed to the repo — scratchpad only, same "one-shot conversion tool" precedent as the 2026-07-02 energy rebalance): for each Pokémon, an HP-bucket table decides how many of its 4 moves land in each cost tier (≤230 HP → 2×tier1+1×tier2+1×tier3+0×tier4; 231-270 → 1+2+1+0; 271-310 → 1+1+1+1; >310 → 0+1+1+2 — lower HP means more cheap/spammy moves, higher HP means more of the expensive top-tier). Within a tier, dmg/cost values are interpolated by each Pokémon's **HP percentile rank within its own bucket** (not by original move power, since the whole point this time was an HP-driven rebalance) plus a small ±12% jitter so same-HP Pokémon don't get numerically identical moves. The pre-existing move **names, types, and any already-set `status` field are preserved** for tier1-3 (only dmg/cost/megaBoost/bonusEnergy are new) — tier4 always gets a **fresh** status-or-selfHeal roll that replaces whatever status the original move had (never both at once — a move can have `status` XOR `selfHeal`, never neither-nor-both stacked). Exactly one of the 4 slots (chosen independently of tier, uniformly random) gets fully overwritten with a support move — this can and does sometimes overwrite a tier4/tier1/etc. slot, so **don't assume a Pokémon's tier-count distribution is fully visible in its final 4 moves** — one tier's worth silently became a support move instead for that specific Pokémon.

**The 8 support moves** — all execute through `executeSupportMove()` (`pokemon_battle.html`) / `executeSupportMoveSrv()` (`server.js`), a parallel dispatch path alongside `doAttack()` triggered when `atk.support` is true (checked at the exact call site that would otherwise call `doAttack`, in `playerAttack`/`cpuAttack` and the PvP `attack` handler) — energy is deducted and the turn ends exactly like a normal attack, just no damage formula runs:
- **撐住 (brace, cost 5)**: sets `G[side+'Braced']`. Checked in `doAttack` right after the `sturdy` ability check — but unlike `sturdy`, **works at any HP, not just full HP** (explicit user requirement, confirmed because the literal Chinese wording didn't require full HP the way the real ability does). Consumed by the very next hit taken (`damage > 0`), whether or not it actually saved the Pokémon.
- **劍舞 (sword-dance, cost 4)**: `aBuff.atkMult = Math.max(aBuff.atkMult, 1.5)` — reuses the exact existing buff mechanism/lifecycle as 全力出擊 (all-out-attack supporter card), so it needs zero new expiry code; persists until the Pokémon's own next attack lands, same as every other `atkMult` source in this game.
- **小偷 (thief, cost 5)**: byte-for-byte reuse of the existing 掠奪(plunder) trainer-card logic (random steal from opponent's hand, respects the 5-card discard cap) — just triggered from a moveslot instead of a hand card.
- **影舞 (shadow-dance, cost 5)**: sets `G[side+'CoinShield']`. Checked in `doAttack` right where `damage` is computed (before `sturdy`/`Braced`) — 50/50 coin flip, heads zeroes `damage` to 0 entirely (full negation, not a partial reduction like `shield`). Consumed by the next incoming attack regardless of the flip's outcome.
- **施加負面效果 (debuff, cost 3)**: the move's data already has `status:{effect, chance:1}` baked in at generation time (one specific status, randomly chosen per-Pokémon when the moveset was generated — not re-rolled at cast time) — 100% chance since there's no damage to trade off against. Single-player calls the existing `tryInflictStatus()` directly; server.js's version is a trimmed inline copy of `doAttack`'s `rollStatus` closure (that closure isn't a standalone function, so it couldn't be reused verbatim).
- **冥想 (meditate, cost 3)**: `G[side+'BonusItemDrawsNextTurn'] += 2` — consumed in `drawAfterTurn`/`drawForRole` by pushing 2 extra cards from the exact same `itemsOnly` pool already used for the normal per-turn draw.
- **詭計 (trick, cost 3)**: `G[side+'BonusSupporterDrawNextTurn'] = true` — consumed the same way, but draws from `TRAINERS.filter(c => c.cat === 'supporter')` (the same pool `standby` already uses). **This is a deliberate, explicit exception to the game's otherwise-universal "supporter cards only ever appear in the starting hand, never redrawn" rule** — confirmed with the user before implementing, since it's the one card in the game that can hand a player a *second* supporter mid-battle.
- **集氣 (focus-energy, cost 3)**: `G[side+'BonusEnergyNextTurn'] += 9` — same mechanism as tier2/3's `bonusEnergy` field, just a fixed +9 instead of move-derived.

**Reflect-style expiry, applied to `Braced` and `CoinShield`**: both are one-shot defensive flags meant to protect against the *opponent's next attack* — so exactly like `reflect`, they must be cleared if the opponent's turn ends *without* them actually attacking (switch, standby, or a status-skip), or they'd persist indefinitely across unrelated future turns. Added `G[op+'Braced'] = false; G[op+'CoinShield'] = false;` alongside every existing `G[op+'Buff'].reflect = false` site — single-player has 5 such sites, server.js has 4 (grep `reflect = false` in either file to find all of them; if you add a 6th/5th way a turn can end without attacking, it needs this triplet too, not just the reflect line).

**CPU AI**: `cpuAttack()`'s `affordable` move filter now also excludes `atk.support` moves entirely (`!cpu.attacks[i].support && ...`) — support moves need situational judgment ("should I brace now, or attack?") that doesn't fit the existing greedy damage-maximizing CPU logic, so CPU simply never uses any of the 8, same "complex-decision cards default to CPU-unused" precedent as 搏命/瘋狂博士. This is single-player only; PvP has no CPU AI to guard.

**Verification approach**: single-player mechanics were verified via direct `javascript_tool` injection (constructing synthetic Pokémon/attacks and calling `doAttack`/`executeSupportMove`/`drawAfterTurn` directly) — note that rapid-fire sequential calls without awaiting each one's internal `setTimeout` chains to fully resolve will cause state bleed between "iterations" that looks like a bug but isn't (observed firsthand: a selfHeal test showed the wrong healed value once because a previous test's queued animation callback fired mid-assertion — reloading the page between isolated tests avoids this). PvP mechanics were verified two ways: (1) extracting the actual `doAttack`/`executeSupportMoveSrv`/`drawForRole` source out of `server.js` via the same regex-extract-and-eval technique `scripts/verify.js` already uses for `POKEMON`/`TRAINERS`, then asserting exact outputs against hand-crafted Pokémon/moves (18 assertions, all passing against the real shipped code — remember to give the attacker and move *different* elemental types in tests that check exact damage numbers, or STAB silently multiplies your expected value by 1.5 and the test fails for the wrong reason); (2) one full real 2-tab WS-driven match confirming 劍舞's buff survives an intervening opponent turn and multiplies correctly into the next real attack's server-computed damage (12 base × 1.5 sword-dance × 1.5 STAB = 27, exactly matching).

## Second moveset rebalance pass (2026-07-15) — tier ranges/HP table changed, mechanics didn't

One day after the 2026-07-14 overhaul above, the user asked for a full re-rebalance with a new rule set — same underlying mechanics (`megaBoost`/`bonusEnergy`/`selfHeal`/`support`+`effect` fields, the `Braced`/`CoinShield`/`BonusEnergyNextTurn`/etc. G-flags, all 8 support move names) but **different numeric tiers, a different HP-bucket table, new support-move costs, and one brand-new rule** — regenerated with a second one-shot script, this time explicitly instructed by the user not to ask clarifying questions and to commit+push autonomously once done.

**New tier table (3 tiers now, not 4 — the old 16-20 cost tier4 was removed)**:

| Tier | Cost | Dmg | Extra effect |
|---|---|---|---|
| t1 | 0-5 | 10-40 | `megaBoost:true` **and** `bonusEnergy` (4-8) — both together now, not split across two tiers like before |
| t2 | 6-10 | 50-75 | `megaBoost:true` **and** `bonusEnergy` (4-8) — same as t1, just pricier |
| t3 | 11-15 | 90-120 | `status` XOR `selfHeal` (unchanged from before) |

**`bonusEnergy` is now derived from the move's own cost position within its tier, not from the Pokémon's HP percentile** — per the user's explicit "依據消耗" (based on cost) wording this time. `dmg`/`cost` themselves are still HP-percentile-driven exactly as before; only `bonusEnergy` changed its interpolation basis, computed as a second, independent lerp: `costPct = (cost - tierCostLo) / (tierCostHi - tierCostLo); bonusEnergy = lerp(bonusEnergyRange, costPct)`.

**New HP→tier-count table** (still 4 buckets, boundaries unchanged, only the counts per bucket changed since there are 3 tiers instead of 4 now):

| HP bucket | t1 | t2 | t3 |
|---|---|---|---|
| ≤230 | 2 | 1 | 1 |
| 231-270 | 1 | 2 | 1 |
| 271-310 | 1 | 1 | 2 |
| >310 | 0 | 1 | 3 |

**New rule: Pokémon with HP > 280 get zero support moves** (all 4 slots are damage moves) — previously every single Pokémon got exactly one. 280 sits inside the 271-310 tier-bucket but is a completely independent threshold with no relationship to the bucket boundaries — don't assume they need to line up. Roughly 94/153 Pokémon kept a support move, 59 lost it entirely.

**New support-move costs** (down from before): 撐住 5 (unchanged), 劍舞 4 (unchanged), 小偷 5 (unchanged), 影舞 5→**2**, 施加負面效果 3→**1**, 冥想/詭計/集氣 3 (unchanged).

**New rule: every support move grants a universal +5 bonus energy next turn, on top of whatever its own specific effect already does** — implemented as one extra line at the very end of `executeSupportMove`/`executeSupportMoveSrv` (after the `switch` block, unconditional): `G[side+'BonusEnergyNextTurn'] = (G[side+'BonusEnergyNextTurn']||0) + 5`. This **stacks additively** with 集氣's own `bonusEnergy:9`, so 集氣 now grants **+14** total, not +9 — `SUPPORT_EFFECT_ZH`'s display strings were updated accordingly in both engines (every entry now says "+下回合能量+5" except focus-energy, which just says the pre-combined "+14" directly rather than "+9（+下回合能量+5）" for readability). **If you add a 9th support move later, remember this universal +5 applies automatically since it's added after the switch, not per-case — you don't need to add it to the new case yourself, but you do need to know it's already there so you don't double-add it.**

**Regeneration methodology difference from the first pass**: rather than building on top of the already-rebalanced (2026-07-14) data — which would have compounded data loss, since that data already has one move slot per Pokémon overwritten with a support-move name/type instead of a real elemental attack name — the script pulled genuine pre-rebalance move flavor (names/types/original status chances) from git history (`git show <commit-before-2026-07-14>:server.js`), and re-applied it against the *current* HP/ability/type/mega fields (which reflect all the individual per-mon tweaks made since, e.g. the Xerneas HP fix). **If you ever need to regenerate movesets a third time, do the same** — reach back to the last commit with genuine hand-authored move flavor rather than the most recent rebalance's output, or you'll be re-randomizing on top of already-lossy data and the "preserve original move names" property silently degrades a little further with each pass.

Verified the same two ways as the first pass: a semantic Node validator asserting the new tier ranges/HP-bucket counts/support-cost table/>280-cutoff hold across all 153 entries, plus extracting `executeSupportMoveSrv`/`drawForRole` from the real `server.js` to assert the universal +5 stacks correctly with each of the 8 effects (9 assertions, all passing) and that `drawForRole` applies the combined total correctly. One live single-player browser check confirmed a HP-≤280 Pokémon's 集氣 move showing "+14" in the UI and setting `G.playerBonusEnergyNextTurn = 14` after actually being cast.

## Weak-multiplier-to-flat-bonus rebalance (2026-07-22)

User reported matches dragging on too long, suspecting weak buff cards were part of the cause. Root cause: an earlier "全面傷害倍率壓縮" pass compressed every card/ability damage multiplier by 80% (raw intended value squashed toward 1.0 via something equivalent to `1 + (raw-1)*0.2`), which had already been baked into the literal numbers in the code (not computed live via `compressMult()`) — so a card meant to swing a fight (e.g. originally ×1.4) had shipped as ×1.08, all but invisible. Flat `atkBonus` cards were never touched by that pass, so they stayed fully effective — meaning the only cards/abilities that actually mattered post-compression were the flat ones, and any buff-card draw with a %-based effect under ×1.1 was effectively a wasted card slot.

**Rule going forward: any self-boosting multiplier below ×1.1 should be a flat `+40` damage bonus instead, not a multiplier.** Multipliers at or above ×1.1 (STAB, blaze-boost, technician, adaptability, colosseum, ocean, gamble, etc.) stay as multipliers — this only applies to the "supposed to help but too weak to notice" band. Debuffs applied to the *opponent* (intimidate ×0.9, chance-debuff ×0.9, thick-fat/solid-rock/frisk-ward/multiscale's damage-taken reductions, stadium field debuffs like rock-field ×0.97) are a different side of the formula (damage-received mitigation, not damage-dealt buff) and are explicitly **not** in scope for this conversion — don't touch values below 1 that are reducing incoming damage.

**Mechanism**: `doAttack()`'s final formula used to be one big product: `(atk.dmg + aBuff.atkBonus + stadiumBonus + reversalBonus) * effectiveAtkMult * ... * abilityDmgMult * ... * megaBoostMult * stadiumMult * typeBoostMult`. Converting a multiplicative term to a flat bonus means moving it out of the multiplication chain and into the additive sum at the front, following the existing `stadiumBonus`/`reversalBonus` precedent (both already flat additions, added 2026-07-08ish for 特訓場/逆轉鬥技場). The current additive sum is `atk.dmg + aBuff.atkBonus + stadiumFlatBonus + legacyDmgBonus + abilityDmgBonus + megaBoostBonus + typeBoostBonus`, where `stadiumFlatBonus = stadiumBonus + reversalBonus + lavaBonus`. `abilityDmgMult` still exists as a separate multiplicative term for the abilities that stayed ≥1.1 (blaze-boost, technician) — **the ternary chain for abilities now has two separate variables, one flat (`abilityDmgBonus`) and one multiplicative (`abilityDmgMult`), split by whether that ability's value is above or below the ×1.1 line.** If you add a new ability with a damage multiplier, decide which chain it belongs in using that same threshold.

**`typeBoost` (used by 龍之波動/dragon-pulse and 順風/tailwind) needed a dual-mode object** since one stayed multiplicative (dragon-pulse, ×1.12) and one converted (tailwind, was ×1.04): `{ type, mult }` for the multiplicative case, `{ type, bonus }` for the flat case. `doAttack` checks both (`typeBoostMult`/`typeBoostBonus`), defaulting to the neutral value (1 or 0) for whichever field isn't present on that particular buff object. If you add a third type-conditional buff card, pick whichever field matches its intended strength — don't assume `typeBoost` is multiplier-only just because dragon-pulse was first.

**Converted this pass** (both engines + `desc:` text + rules-modal `<li>` text in `pokemon_battle.html`/`public/single.html`/`public/pvp.html`, verified via `grep -noE '×1\.0[2468]'` returning zero non-comment hits afterward): 捨身猛擊/focus-punch (×1.04→+40, self-damage 20% HP kept), 連擊/double-strike (×1.08→+40, `doubleStrike` status-reroll flag kept), 順風/tailwind (×1.04→+40, flying-only), 詭計/trick support move (×1.02→+40), and abilities 毅力‧guts/大力士‧huge-power/硬爪‧tough-claws/背水之刃‧desperate-blade/淬鍊之心‧status-immune-once/機械之心‧item-synergy/海洋支配‧drizzle-ocean/熔岩大地‧drought-lava/指揮‧legacy-boost (all ~×1.02–1.06 → +40), the generic Mega-evolution bonus (×1.02→+40), and 熔岩火山/stadium-lava's fire-type bonus (×1.02→+40; its water-type ×0.9 penalty is untouched, it's a debuff).

**Verification approach**: rather than a full integration test, isolated `doAttack()` in a live single-player browser session (`javascript_tool`) — forced a controlled attacker/defender pair (neutral types, huge HP pool to avoid KO/clamping noise), called `doAttack()` directly with a synthetic 100-power move, and diffed defender HP before/after. Confirmed: baseline 100 dmg with no buffs; +40 exactly with an ability, the Mega bonus, the legacy-boost buff, or a `typeBoost.bonus` all in isolation; an unconverted ability (blaze-boost ×1.1) still stacks multiplicatively with STAB as before (100 → 121, matching 100×1.1×1.1 exactly); the unconverted dragon-pulse `typeBoost.mult` path still works (100×1.12=112); and calling `applyTrainer()` for focus-punch end-to-end sets `atkBonus` to 40 (not `atkMult`) while its 20%-self-damage side effect still fires correctly.
