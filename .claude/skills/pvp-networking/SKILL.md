# PvP Networking (йҖЈз·ҡзӣёй—ң)

## Architecture: server-authoritative, thin client

`server.js` (Node + `ws` WebSocket + Express) owns **all** game state (`room.G`) and **all** randomness. `public/pvp.html` never computes outcomes locally вҖ” it sends an intent message, waits for the server's broadcast, and re-renders from whatever state arrives. If you're ever tempted to have the client predict/compute a result (even just for a snappier UI), don't вҖ” every prior instance of client-side prediction in this codebase was later revealed to be wrong or exploitable. The one narrow exception: the client locks its own UI (`G.busy = true`) optimistically right after sending a message, purely to prevent double-sends before the server's reply arrives вҖ” it doesn't compute or display any *result*.

## Room lifecycle

`rooms` (in-memory `Map`, no persistence вҖ” a server restart drops all active games). Room object: `{code, p1, p2 (ws refs), phase: 'waiting'|'selecting'|'battle'|'done', p1Roster/p2Roster, p1Team/p2Team, p1Ready/p2Ready, p1Rerolls/p2Rerolls, coinFlip, G}`. `ws.on('close', ...)` deletes the room outright if `phase !== 'done'` вҖ” **any** disconnect during team-select deletes the room, even a brief network blip, which is why reroll/other messages sent right after a disconnect must fail gracefully (`send(ws, {type:'error',...})`) instead of the old silent `return` that left the client hanging forever (fixed 2026-07-01).

## Message protocol (client вҶ’ server)

`create_room`, `join_room`, `reroll`, `select_team`, `attack`, `standby`, `switch`, `ko_switch`, `use_trainer`, `discard`, `discard_trade`, `chat`. All routed through one `handleMessage(ws, msg)` function, one big `if (type === '...')` chain (not a lookup table вҖ” grep the literal string when hunting for a handler). Every handler that can fail validation should `send(ws, {type:'error', message:...})` rather than silently `return`ing вҖ” a silent return with no reply is indistinguishable, from the client's perspective, from a dropped packet, and the client has no generic timeout/retry, so it just hangs (this exact bug caused the "йҮҚж–°з”ҹжҲҗжҢүйҲ•еҒ¶зҲҫжІ’еҸҚжҮү" report, fixed 2026-07-01).

Server вҶ’ client: `room_created`, `joined`, `opponent_joined`, `opponent_ready`, `roster_update`, `battle_start`, `update` (the workhorse вҖ” sent after nearly every action, carries `{state: G, log: [...], actor, atkType?}`), `game_over`, `error`, `opponent_disconnected`, `chat`.

## Client state sync

`public/pvp.html`'s `applyServerState(sG)` is the single place server state gets copied into the client's local `G` вҖ” it remaps role-based server fields (`p1Deck`/`p2Deck`) into perspective-based client fields (`playerDeck`/`cpuDeck`) using `myRole`. If you add a new field to the server's `G` object that the client needs to display, it must be added to `applyServerState` too, or it'll silently never reach the UI. `G.busy` is *derived* here (`turn !== me` or an opponent-pending-KO-switch), not sent directly вҖ” don't add a redundant server-sent `busy` field.

## Known fragile spots

- **Reroll button**: fixed 2026-07-01 (see above) вҖ” if similar "button seems unresponsive" reports come in for other actions, check for the same silent-`return`-on-invalid-state pattern first before assuming it's a rendering bug.
- **`pendingKOSwitch` is single-valued** (`null | 'p1' | 'p2'`) but some effects (жҗҸе‘Ҫ, and the double-KO fix) need *both* sides to sequentially replace вҖ” handled via `G.pendingKOSwitchQueue`, drained one at a time inside the `ko_switch` handler. If you add another effect that can KO both sides at once, reuse this queue rather than inventing a second one.
- **Turn-ownership вҖ” FIXED 2026-07-08, this note previously described the bug, not current behavior.** When an attacker dies from their *own* status/reflect damage before their attack lands (`sResult.died` in the `attack` handler), `G.turn` used to be left unchanged, letting that player act again immediately after replacing their fainted mon. Both the `sResult.died` branch and the sibling reflect-death branch now explicitly set `G.turn = op; G.round++; drawForRole(G, op);`, matching every other KO path. See battle-logic skill's KO/double-KO section for the full fix writeup вҖ” if you find a *third* "attacker dies without the defender dying" branch anywhere, verify it sets `G.turn = op` too, this omission has already recurred once.
- Every WS handler should stay wrapped so one bad message can't crash the whole process and disconnect both players вҖ” keep new handlers inside the existing try/catch scope in `wss.on('connection', ...)`.

## Deployment (Zeabur)

`server.js` via `npm start`; `PORT` auto-injected. PostgreSQL auto-injects `POSTGRES_*` env vars when in the same Zeabur project вҖ” **do not manually set `DATABASE_URL`**, a prior attempt to do so broke the Docker build (space-containing connection string). No DB tables are actually required for gameplay (`rooms` is in-memory); the `pool` connection is best-effort and the server logs "No DB configured, running in-memory only" and continues fine without it вҖ” useful to know when testing locally with `node server.js` and no Postgres available.

## Accounts / persistence (2026-07-10)

Login is **purely additive**, never a gate on PvP. `wss.on('connection', (ws, req) => {...})` parses an optional `?token=` query param; a valid token sets `ws.userId`/`ws.username`, an invalid/missing token or absent `pool` just leaves them `undefined` and the connection proceeds exactly like today's anonymous flow вҖ” **never reject a WS connection over auth**. Token verification is async (`pool.query`), so incoming messages are queued (`msgQueue`/`authPending` in the connection handler) until it resolves, so a fast first message right after connecting isn't dropped.

3 Postgres tables (`users`, `teams`, `weekly_stats`), all `CREATE TABLE IF NOT EXISTS` in `initDB()` вҖ” never `DROP`/destructive `ALTER`, this data must survive every deploy. `teams.pokemon_ids` is a persistent 6-PokГ©mon pool per account (not a fixed 3-team) вҖ” every match still goes through the ordinary `select_team` (pick 3-of-6) flow unchanged, so this required zero changes to `battle_start` triggering logic. `randomRoster(6, 300, 1)` (the same helper anonymous players use) is wrapped by `generatePlayerPool()` for all three places a 6-pool gets generated: registration, `edit_team` candidates, and stale-id auto-repair (`loadUserTeam()`/inline repair in `create_room`/`join_room`) вҖ” **the `hpCap=300, maxAtCap=1` params are a deliberate balance rule the user asked for, not a default to tweak casually**.

**Same вүҘ300-HP cap, enforced again at the 3-selection step (2026-07-13)**: `randomRoster`'s `maxAtCap=1` only constrains how the *6-pool itself* gets generated вҖ” it says nothing about the 3-of-6 pick a player makes for an actual battle. The user explicitly asked for a second, independent guard here: *even if* a 6-pool somehow ends up with 2+ PokГ©mon at HPвүҘ300 (a persistent account pool predating this rule, a future roster-gen change, etc. вҖ” "еҰӮжһңйҡҠдјҚжңүдәҶжІ’й—ңдҝӮ" / "if the pool already has it that's fine"), the player still can't put more than one of them into their actual 3-PokГ©mon battle team. Implemented in 3 places, none sharing code (single-file-per-engine, same as `effectiveCost`'s 3-copy situation above): `pokemon_battle.html`/`public/single.html`'s `showTeamSelectScreen()` (client-only, no server to double-check against, since single-player has no adversarial client) blocks the click and shows an inline error under the card count; `public/pvp.html`'s `toggleTSCard()` does the identical UI-side block; **`server.js`'s `select_team` handler also validates independently** (`selected.filter(p => p.hp >= 300).length > 1` вҶ’ reject with an error, never trust the client's own count) вҖ” this one matters for real, since a spoofed WS client could otherwise submit indices bypassing the UI block entirely. All three use `hp >= 300` (not strict `>`), matching the existing `hpCap` semantics used everywhere else in this codebase вҖ” 16 of the 153 PokГ©mon have exactly `hp:300`, so `>` vs `>=` is not a cosmetic distinction here, it changes which mons count.

Roster source in `create_room`/`join_room`: `getRosterForConnection(ws)` вҖ” if `ws.userId && pool`, reads the account pool (with auto-repair); otherwise (anonymous, no DB, or a DB error) falls back to the exact same `randomRoster()` call anonymous players have always used. Any DB failure here degrades to the anonymous path rather than erroring the room.

Logged-in players get an "з·ЁијҜйҡҠдјҚ" button in team-select (replacing, not alongside, the anonymous "йҮҚж–°з”ҹжҲҗ" button) вҖ” `edit_team` WS message generates 6 fresh candidates (costs 1 of 3 uses immediately, whether or not anything gets swapped, same cost model as reroll), `confirm_team_edit` applies validated `{slotIdx, candidatePokemonId}` pairs to both the in-room roster and `teams` (permanent). Server never trusts client-supplied candidate ids вҖ” validates against the most recently generated batch.

`game_over` is broadcast from a single place now: `endGame(room, winner, log, extra={})`, which all 11 former call sites route through. It sets `G.winner`/broadcasts/`room.phase='done'` synchronously, then fires `recordWeeklyStats()` async (never blocks or risks the broadcast) вҖ” upserts `weekly_stats` only for sides with a `userId`; a draw or two-anonymous match writes nothing. Week bucketing is `mondayOfWeek(date)`, no cron вҖ” a new week's row just gets created lazily on the first win that lands in it. **If you ever add a 12th path that can end a game, route it through `endGame()`, not a fresh `broadcast(room, {type:'game_over',...})`** вҖ” this is the same "grep every sibling call site" discipline as `adding-cards`, just applied to `game_over` instead of `TRAINERS`.

GM admin: `users.is_admin`/`disabled` columns, no self-registration вҖ” **to make an account an admin, connect to Postgres directly and run `UPDATE users SET is_admin = true WHERE username = '...';`** (no UI for this, intentionally вҖ” see `project_pokemon_battle.md` memory for the exact recommended command). `requireAdmin` middleware layers on `requireAuth`, guards `/api/admin/*`; `public/admin.html` is a fully separate static page reusing the same `/api/login`+`/api/me`-adjacent auth (it actually confirms admin status via a 403 from `GET /api/admin/users` rather than a dedicated `isAdmin` field, since `/api/me` doesn't expose that field).

**Local Postgres testing (2026-07-10, now set up)**: Homebrew `postgresql@16` is installed on this dev machine (`brew install postgresql@16`), started manually via `pg_ctl -D /opt/homebrew/var/postgresql@16 -l <logfile> start` (not registered as a `brew services` login item вҖ” start/stop it per session). A throwaway `pokemon_battle_test` database was created with `createdb pokemon_battle_test`. To test locally: `DATABASE_URL=postgresql://$(whoami)@localhost:5432/pokemon_battle_test node server.js`.

**SSL gotcha вҖ” two rounds, resolved (2026-07-10)**: the original `new Pool({ connectionString: pgUri, ssl: { rejectUnauthorized: false } })` unconditionally forced an SSL handshake. Round 1 fix only special-cased `localhost`/`127.0.0.1` to skip SSL, keeping it forced for "remote" hosts under the assumption Zeabur's Postgres needed it вҖ” **that assumption was never verified and was wrong**. Deploying it broke registration/login/leaderboard in production with a generic `db_error` on every single DB query (confirmed via Zeabur logs: `PostgreSQL not available, running without DB: The server does not support SSL connections` вҖ” Zeabur's managed Postgres doesn't speak SSL either). This bug had actually existed since the very first DB connection code was written, long before the account system вҖ” it just never surfaced because `rooms` is in-memory and nothing ever issued a real query against it. Final fix: don't force `ssl` at all вҖ” `new Pool({ connectionString: pgUri })`, letting the `pg` driver default to no SSL unless the connection string itself has `?sslmode=require`. **Lesson**: when fixing a connection issue for one environment, don't assume "it currently works" for another environment proves its config is correct вҖ” verify via logs, especially if that code path was never actually exercised before (a `pool` that's "connected" but has never run a real query proves nothing about whether queries will succeed).

Full end-to-end verification against this real DB (2026-07-10) confirmed: all 3 tables + FK/cascade constraints created correctly by `initDB()`; registerвҶ’loginвҶ’`/api/team` round-trips through a real persisted 6-PokГ©mon pool; WS `create_room`/`join_room` for a logged-in user reads that exact pool; stale-id corruption (`UPDATE teams SET pokemon_ids = '{99999,...}'`) triggers auto-repair and overwrites the DB row, both via `GET /api/team` and via WS `create_room`; `edit_team`/`confirm_team_edit` swaps persist permanently across brand-new connections, reject spoofed candidate ids, and correctly cap at 3 uses per room; two full scripted PvP matches (driven via a Node WS client that auto-plays both sides) correctly upserted `weekly_stats` (increment, not overwrite) only for logged-in sides вҖ” a mixed logged-in-vs-anonymous match wrote nothing for the anonymous side; `GET /api/leaderboard` reflects real data sorted by wins; the full GM admin flow (mark-admin via direct SQL вҶ’ `public/admin.html` login вҶ’ disable/re-enable вҶ’ reset-password invalidating the old password вҶ’ edit weekly stats reflected live in the UI вҶ’ hard-delete cascading to `teams`/`weekly_stats`) all worked exactly as designed. **This account system is now genuinely production-verified, not just logic-verified.**

## Leaderboard tie-break by win rate (2026-07-13)

`GET /api/leaderboard` sorts `ORDER BY ws.wins DESC, win_rate DESC` вҖ” `win_rate` is computed in SQL (`wins::float / (wins+losses)`, `0` when no games) rather than client-side, so the tie-break is consistent regardless of who's asking. Verified against the real local Postgres DB: two accounts with equal win counts (5-5 vs 5-1) correctly rank the higher-win-rate one first.

## Spectator system (2026-07-13)

A third connection sending `join_room` with a code whose room already has both `p1` and `p2` no longer gets the `жҲҝй–“е·Іж»ҝ` error вҖ” it's added to `room.spectators` (array of ws refs, initialized empty in `create_room`) and `ws.role = 'spectator'`. This single architectural fact made the whole feature cheap: **`battle_start`/`update`/`game_over`/`chat` were already centralized through one `broadcast(room, msg)` function**, so adding `for (const s of room.spectators) send(s, msg)` there is the one change that makes every existing game event reach spectators automatically вҖ” no per-message-type edits needed anywhere else server-side.

Two places needed explicit spectator-awareness because they don't go through `broadcast()`/assume exactly two roles:
- `handleMessage`: added `if (role === 'spectator') return;` right after `const role = ws.role`, so a spectator's stray messages (if the client-side guard below is ever bypassed) are silently ignored rather than mis-handled by whatever `if (role === 'p1') ... else ...` branch happens to run next.
- `ws.on('close', ...)`: previously assumed every disconnecting socket was `p1` or `p2` and would (a) notify "the other side" and (b) delete the room if the game wasn't `'done'` yet. A spectator disconnecting used to satisfy `ws.role !== 'p1'` and fall into the `p2`-disconnect branch вҖ” **wrongly nuking the room out from under two players who were still mid-game**. Fixed by checking `ws.role === 'spectator'` first and just removing it from `room.spectators`, no room deletion, no bogus `opponent_disconnected`.

Bootstrapping a spectator's view depends on `room.phase` at join time: if a battle is already running (`'battle'` or `'done'`), they get `{type:'spectate_joined', phase, state: room.G}` and the client jumps straight to the battle screen; if the game hasn't started yet (`'waiting'`/`'selecting'`), they get `phase` with no `state` and the client shows a `spectate-wait-screen`, then the *next* real `battle_start` broadcast (now reaching spectators automatically) carries them into the battle screen along with everyone else.

Client-side (`public/pvp.html`): spectators fix `myRole = 'p1'` (never their own role вҖ” they don't have one) purely so `applyServerState`'s existing `me`/`op` remapping renders p1 as the "player" panel and p2 as "cpu" вҖ” a stable viewing angle regardless of who's actually attacking. A global `isSpectator` flag then does double duty: (1) a defense-in-depth guard inside `send()` that drops every outgoing message once set, and (2) forces every action affordance (attack buttons, mega, standby, trade, hand-card use buttons, bench switch-clicking) to `disabled` regardless of what the normal G.busy/energy logic would compute вҖ” **both layers matter**, because disabling only the network layer would still let a spectator click a button that mutates local-only state (e.g. `playerAttack()` sets `G.busy=true` optimistically before ever calling `send()`) with no server reply ever arriving to correct it. Anywhere the existing code branches on `myRole` directly (KO-switch popup trigger, discard-modal trigger, coin-flip text, result-screen "you won/lost" text) also needed an `!isSpectator` guard, since `myRole==='p1'` fixed-orientation would otherwise make the client think the spectator personally needs to pick a KO replacement, discard, etc.

**Known accepted trade-off, not a bug**: spectators see full server state including both players' hidden hand contents (`G.playerHand`/`G.cpuHand` both resolve to real hands, just relabeled). There's no sanitized/redacted broadcast for spectators. Fine for a casual party game; would need a real fix (a separate hand-scrubbed message for the spectator audience) if this is ever used somewhere collusion between a spectator and a player is a real concern.

Verified against a real 3-tab run (2 players + 1 spectator) via `javascript_tool`: spectator joining before team-select saw the wait screen then auto-transitioned on `battle_start`; joining after battle start (separately tested) delivers state immediately; a real attack from p1 correctly updated the spectator's HP/log in real time; spectator's own attack button stayed disabled throughout; closing the spectator's WS did not affect the two players' ongoing game (attack still worked, turn still advanced normally afterward) вҖ” confirming the `ws.on('close')` fix actually prevents the room-deletion bug described above.

## Turn-start browser notification (2026-07-13)

`public/pvp.html` requests `Notification` permission once, lazily, inside `case 'battle_start':` (`if (Notification.permission==='default') Notification.requestPermission()`) вҖ” not on page load, so the browser's permission prompt only ever appears in the context of an actual match starting, not the moment someone opens the lobby. The actual notify call is `notifyIfMyTurnStarted(wasMyTurn, isMyTurn)`, invoked from `case 'update':` right after `applyServerState()`, comparing `!G.busy` *before* vs *after* applying the new state вҖ” it only fires on the falseвҶ’true transition (turn just started, not "still my turn" on every subsequent unrelated update), and only if `Notification.permission==='granted'` and `document.hasFocus()` is false (tab backgrounded/unfocused) вҖ” checking `document.hidden` instead would have been wrong, since a visible-but-unfocused tab (e.g. user alt-tabbed to another app while pvp.html stays the frontmost tab) is exactly the case that most needs a notification and `document.hidden` wouldn't catch it. Uses `tag:'pvp-turn'` so a player who lets multiple turns pass while away gets one replaced notification, not a stack. Verified via two approaches: (1) isolated unit-style calls with a mocked `Notification`/`document.hasFocus` covering all 4 branches (transition+unfocused вҶ’ fires; already-my-turn вҶ’ no-op; still-not-my-turn вҶ’ no-op; transition+focused вҶ’ no-op); (2) a real 2-tab PvP match with `document.hasFocus` monkey-patched to `false` on one tab, confirming the real gameplay turn-flip actually invokes it end-to-end, not just the isolated function.
