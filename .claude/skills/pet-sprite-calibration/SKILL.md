# Pet sprite calibration — animation scale + companion tracking

`public/tamagotchi.html`'s pet stage composites overlays (a companion sprite, action-animation sprite sheets) on top of `#pet-sprite`. Every pixel position or size here is a cross-referencing problem between **multiple different art sources drawn at different native resolutions and different poses**, and most mistakes so far have come from calibrating against the wrong reference, not from imprecise reading. Read this whole file before touching `ANIM_DISPLAY_TARGET`, `PET_ANIM`, or the companion orbit logic.

## Animation display scale (Eat/Sleep/Attack sizing)

`playPetAnim()` overlays a PMDCollab sprite-sheet frame — a **separate art source** from the idle GIF, with its own native resolution per animation per species (see `PET_ANIM` for the raw `[w,h,frames]`). Displaying these at native pixel size makes them look wildly smaller than the idle sprite (e.g. Pikachu's Sleep frame is 32×40 natively, vs. the idle GIF nearly filling the full 96×96 CSS box).

**Scale each animation independently against its own frame size — do not derive one shared scale per species.** This was tried twice and reverted both times:

- v1 was unscaled (native pixel size) → Sleep/Eat looked tiny next to idle.
- v2 computed **one scale per species** from PMDCollab's own Idle-frame size, reasoning that "PMDCollab keeps a consistent character scale across its own rig, so one multiplier should work for Eat/Sleep/Attack too." This is theoretically tidy but wrong in practice: PMDCollab's Attack frame canvas is genuinely 40–75% bigger than its Idle frame (room for an outstretched arm/effect, not a bigger character), so the *same* multiplier that makes Idle-equivalent look right makes Attack noticeably bigger than Sleep, and neither one reliably lands close to the real 96×96 idle box. The user reported oversized Attack and undersized Sleep as two *separate* bugs before this was diagnosed — both were the same root cause.

**Current (v3, correct) approach**: compute `currentAnimScale = ANIM_DISPLAY_TARGET / Math.max(w, h)` **fresh for every `playPetAnim()` call**, using that specific animation's own `[w,h]` from `PET_ANIM`. `ANIM_DISPLAY_TARGET = 88`, tuned empirically (not derived) by triggering Eat/Sleep/Attack on Pikachu and comparing `getBoundingClientRect()` output against the idle sprite's `96×96`:

```
Eat    [24,48] → 44×88
Sleep  [32,40] → 70×88
Attack [80,80] → 88×88
idle (unrelated to this formula, fixed CSS box) → 96×96
```

All three land in the 70–96px range now, visually close to each other and to idle — this is what "look like they're about the same size" actually requires, not literal PMDCollab-rig fidelity. The trade-off: switching between animations on the *same* pet produces a small visible size jump at the switch instant (since each animation targets 88px independently rather than sharing a species-derived multiplier) — this reads as normal "different pose, different silhouette" variation, not as a bug, and is far less noticeable than one animation being persistently much smaller than idle.

If `ANIM_DISPLAY_TARGET` needs retuning later (e.g. after adding a species whose animations still look off), don't derive it mathematically — trigger the animation, read `document.getElementById('pet-sprite').getBoundingClientRect()`, and adjust the constant until it's close to `96`. Empirical testing caught both of the wrong turns above; formula-only reasoning didn't.

## Preloading (avoid the "pet disappears" bug)

`playPetAnim()` must **preload** the PMDCollab sprite-sheet URL with a throwaway `new Image()` and only swap `#pet-sprite`'s `src` to the transparent placeholder inside that image's `onload` callback. If you swap to the transparent placeholder immediately and let `background-image` load asynchronously in the background, there's a window — invisible on a fast connection, very visible on a slow one — where the transparent placeholder is showing and the background sprite sheet hasn't painted yet, making the pet appear to vanish entirely. This shipped as a real bug (reported as "餵食的畫面沒有出來，寶可夢會不見") masked for a while by an unrelated bug where the "transparent" placeholder was accidentally opaque white — once that one was fixed, the underlying load-race became visible as true invisibility instead of a wrong-color flash. On `onerror`, abandon the animation attempt entirely and leave the idle sprite untouched — never leave the pet mid-transition with no visible image.

## Companion sprite (Pichu, orbiting)

`startCompanion()` renders a small 16×16-native pixel-art Pichu (`PICHU_COMPANION_FRAMES`, 2 bounce frames) that orbits `#pet-sprite` continuously via `requestAnimationFrame`.

**Critical gotcha, already hit once**: `#pet-sprite` has `transition:left 1.8s ease-in-out, top 1.8s ease-in-out` for smooth walking. That means `img.style.left`/`img.style.top` (the inline style *attribute*) update **instantly** the moment JS sets a new target, but the actual **painted position** eases toward that target over 1.8 seconds. If the orbit loop reads `parseFloat(img.style.left)` as the pet's "current" position, it's actually reading the *end-of-walk target*, not where the pet visually is right now — for up to 1.8s after every walk step, the companion orbits a point the pet hasn't reached yet, and the two visibly drift apart (confirmed by screenshot: companion ended up on the opposite side of the room from the pet). **Always read position via `img.getBoundingClientRect()`, converted into stage-local coordinates (`rect.left - stageRect.left`), never via `img.style.left`/`top` directly**, whenever something needs the pet's true current on-screen position while `#pet-sprite` might be mid-transition. This applies to any future feature that needs to track the pet's live position, not just the companion.

Bonus effect of using `getBoundingClientRect()`: it also reflects the pet's *current rendered size* (post `transform:scale()`), so the orbit radius automatically stays proportional even while an Eat/Sleep/Attack animation has temporarily resized `#pet-sprite` — no separate handling needed for that case.

Asset source: `Rangi42/polishedcrystal` (an open-source Pokémon Crystal romhack disassembly on GitHub), `gfx/minis/{species}.png` — these are the game's own tiny "mini sprite" icons (16×16 per frame, 2 frames stacked as 16×32), used in-game for the Pokédex/party menu. They ship as flat grayscale (4-shade Game Boy Color palette: 0/85/170/255), not colorized to the species' real colors — recolor manually (map the 4 gray levels to species colors, `255→transparent`) before using, or the sprite will just look like a monochrome menu icon rather than a recognizable Pichu. The repo has no explicit LICENSE file; treated the same as this project's other non-formally-licensed fan Pokémon sprite sources (PokeAPI/Showdown/PMDCollab) — fetch/derive with a visible attribution line, consistent with existing practice, not a novel exception.

## Idle/Walk unified onto PMDCollab too (2026-07-19, supersedes the AI-art pilot below)

The root cause of "素材不夠統一" was never that the 13 species' art styles clashed with each other — it was that **idle/walking used a completely different source (PokeAPI Gen5 B/W animated GIF) than Eat/Sleep/Attack (PMDCollab)**. PMDCollab's own `AnimData.xml` already ships `Idle` and `Walk` entries for every species (confirmed via `curl` against all 13 dex ids — every one has both), in the exact same art pipeline already trusted for Eat/Sleep/Attack. Switching idle/walk to also pull from PMDCollab unifies all five states for all 13 species with **zero new generation, zero copyright exposure beyond what's already accepted**, and no dependency on ChatGPT (which was hitting content-policy refusals on later species — see the AI-art section below for why).

- `PET_ANIM[speciesId]` now carries `Idle:[w,h,n]` and `Walk:[w,h,n]` alongside `Eat`/`Sleep`/`Attack`.
- `startStage()` sets `#pet-sprite`'s `src` to the static PokeAPI/Showdown PNG only as an instant-visible placeholder before `playPetAnim(speciesId,'Idle',true)` upgrades it — this avoids a blank pet while the first PMDCollab frame is preloading.
- `walkToRandomSpot()` now also triggers `playPetAnim(speciesId,'Walk',true)` for the duration of the CSS position transition (a `setTimeout` matching the existing `1.8s` `left`/`top` transition, not a `transitionend` listener — simpler, and this project already uses fixed-duration timers elsewhere for the same reason), then reverts to `Idle` loop. Both checks guard on `!sleeping` so a stray walk tick can't interrupt an active Sleep loop.
- `stopPetAnim()`'s "return to neutral" step is now `playPetAnim(speciesId,'Idle',true)` instead of resetting `img.src` back to the PokeAPI GIF — there's no more "neutral static image" state, the pet is always inside some PMDCollab loop (Idle, Walk, or Sleep) once the stage has loaded.
- `spriteUrls()` (PokeAPI/Showdown static+animated) is **not removed** — it's still used for the species-picker grid thumbnails and the read-only 拜訪朋友 mini-room, contexts where the old inconsistency was never actually visible/relevant. Only the *live, interactive* pet stage was in scope.

## AI-generated static pose art (`PET_ART_OVERRIDE`) — retired in favor of the unified PMDCollab approach above

**Status as of 2026-07-19: `PET_ART_OVERRIDE` is empty.** The Pikachu pilot below (Sleep/Eat/Attack as AI-generated static poses) got superseded the same day it shipped, once the "Idle/Walk unified onto PMDCollab too" fix above made every state for every species come from one source — keeping the AI poses for just Pikachu's 3 action states would have re-introduced the exact inconsistency being solved (Pikachu's idle/walk = PMDCollab, but its Sleep/Eat/Attack = a different, softer-shaded art style). The mechanism (`PET_ART_OVERRIDE` object, `playStaticPoseArt()`, the branch check at the top of `playPetAnim()`) is left in place, not deleted, in case a future request supplies a **complete** matching set (not just 3 poses) worth switching a species over to.

**Also worth knowing if AI generation gets revisited**: mid-session, further species requests started getting blocked by ChatGPT's own "third-party content similarity" safeguard. The dividing line, confirmed empirically: **naming a Pokémon species in a text-only prompt** (e.g. "generate 妙蛙種子 in this style") reliably triggers the block; **uploading a reference image and asking for a style-transfer/analysis with no character name in the text** (the technique that produced the Pikachu art) does not. If reviving this path, always attach a reference image (the project's own `PokeAPI`/`Showdown` static sprites work fine as references — genuinely the same images already used elsewhere in this game) rather than describing the species by name in text, and avoid the word "Pokémon" even in the reused Style Specification text (it's plausible that alone raises risk, though unconfirmed) — describe the target format generically ("GBA-era JRPG chibi pixel sprite") instead.

As of 2026-07-19, individual species can get a per-state art override that replaces the PMDCollab sprite-sheet source for `Sleep`/`Eat`/`Attack` — Pikachu (25) is the pilot, sourced from the user's own ChatGPT session (`參考圖檔/皮卡丘_寵物分鏡.png`, a 1254×1254 reference sheet with ~30 poses grouped by category, generated via a "style-extraction" technique: upload a real Pikachu sprite → ask ChatGPT to describe it as an abstract technical **Style Specification** — GBA sprite style, 12–16 color palette, no anti-aliasing, 1px outline, top-left lighting, chibi proportions — with no character name in the prompt at all → reuse that spec to generate new poses). This is categorically different from the PMDCollab path and needs its own handling, not a drop-in replacement:

1. **These are single static illustrations, not sprite-sheet frames.** No walk-cycle, no multi-frame breakdown — one pose per state. `playPetAnim()` therefore branches at the top: if `PET_ART_OVERRIDE[speciesId][animName]` exists, it calls `playStaticPoseArt()` instead of the PMDCollab `reveal()` path — no `background-image`/`steps()` CSS animation, no `--frames`/`--strip-w`, just a direct `img.src` swap held for a fixed `STATIC_POSE_MS` (1600ms) then reverted via the same `stopPetAnim()` both paths already share.
2. **Idle/walking deliberately still uses the original animated GIF, not this art**, per explicit user decision (asked via AskUserQuestion 2026-07-19) — the reference sheet has no walk-cycle poses, so fully replacing idle would turn the continuous wandering animation into a static image sliding across the screen. Scope AI-art overrides to the discrete action states only (`Sleep`/`Eat`/`Attack`) unless a future reference sheet specifically includes a multi-frame walk.
3. **Prep pipeline** (source sheet → embeddable asset), all done with PIL, not manually eyeballed:
   - The reference sheet has **zero alpha channel** (flat RGB, solid dark-navy background, e.g. `(15,20,37)`) — crop the target pose's bounding box first (auto-detect via row/column background-fraction scanning: a row/column is "content" if less than ~99.5% of its pixels match the background color within a small tolerance — far more reliable than eyeballing coordinates off a screenshot preview, and this project has been burned by eyeballed-coordinate mistakes before, see the retired wearables section below), **then** chroma-key the background to transparent (per-pixel distance from the sampled bg color, threshold ~20–28 out of a 0–765 sum-of-channel-diffs scale worked cleanly here with no halo).
   - Resize (LANCZOS, not nearest-neighbor — this art has soft shading/anti-aliasing unlike PMDCollab's hard pixel edges, so `image-rendering:pixelated` on `#pet-sprite` would look jagged if the browser had to scale it live) so the **longest dimension equals `ANIM_DISPLAY_TARGET` (88px)** at prep time — this means `playStaticPoseArt()` doesn't need to rely on live `transform:scale()` the way the PMDCollab path does; the asset ships pre-sized.
   - Embed as a base64 `data:` URI directly in `PET_ART_OVERRIDE` (small enough per pose, ~10-16KB) rather than a separate file under `public/` — unlike the earlier "don't base64 13 species × multiple states, that'll bloat the HTML" reasoning in the original Phase-3 plan, a single species' 3 poses is small enough that the plan's original file-based approach turned out to be premature; revisit file-based storage only if/when more species are added and the HTML file size actually becomes a problem.
4. **No preload-before-reveal needed for this path** (unlike PMDCollab's `playPetAnim()`, which fetches over the network and must gate the reveal on `onload` to avoid a "pet disappears" flash) — because the art is a `data:` URI already fully resident in memory once the page's JS has parsed, `img.src = art.src` is synchronous with no network round-trip, so there's no loading race to guard against.
5. **Extending to more species**: add a new `PET_ART_OVERRIDE[speciesId]` entry following the same `{ w, h, src }` shape per state; `playPetAnim()`/`spriteUrls()` need no further changes — species without an override key simply keep falling through to the existing PokeAPI/Showdown/PMDCollab sources, so this is an incremental, species-by-species rollout, not a flag day.

## Third override path: ready-made animated GIFs (`PET_GIF_OVERRIDE`) — Fuecoco/呆火鱷 pilot, 2026-07-21

A user can supply a **matched set of already-finished, already-animated GIFs** (not a single reference sheet to slice, not sprite-sheet frames to re-assemble) — this happened for 呆火鱷 (dex 909, `參考圖檔/呆火鱷＿素材/idle.gif`+`eat.gif`+`sleep.gif`+`happy_jump.gif`). This is architecturally distinct from both existing paths and needed a third one, `PET_GIF_OVERRIDE` + `playGifArt()`, checked in `playPetAnim()` between `PET_ART_OVERRIDE` and the `PET_ANIM`/PMDCollab fallback:

**How to tell which of the three paths a new asset needs:**
- Sprite-sheet strip with N frames of a fixed size, needs `steps()` CSS animation to play → **PMDCollab path** (`PET_ANIM`), or if it's a *new* remote sprite-sheet source (not PMDCollab), you'd need to extend `playPetAnim()`'s reveal() itself, not add a new path.
- Single static illustration, one per state, no animation of its own → **`PET_ART_OVERRIDE`** (`playStaticPoseArt()`), held for a fixed `STATIC_POSE_MS`.
- Already a real animated file (GIF/APNG/WebP) that the browser can loop natively on its own → **`PET_GIF_OVERRIDE`** (`playGifArt()`) — just point `img.src` at it and let the browser decode+loop it; no `steps()`, no per-frame math.

**Asset storage — file, not base64, past a certain size.** These GIFs were 37-82KB each (vs. the Pikachu static poses' 10-16KB, which made base64-in-HTML reasonable). Copied to `public/pet-art/{speciesId}/{name}.gif` and served via the existing `express.static('public')` (same pattern already used for `public/badges/`) instead of inflating `tamagotchi.html` by ~130% base64 overhead × 4 files. If a future GIF set is small enough to comfortably base64 (rare — real animated GIFs are usually much bigger than a single static pose), that's still a valid choice; the deciding factor is file size at the point of integration, not a hard rule against either approach.

**Preload-before-reveal is still required**, even though the file is same-origin — `img.src` pointing at any URL still means a real (if fast) network round-trip before the browser has bytes to paint, and skipping the preload-gate here would reintroduce the exact "pet disappears" bug documented above for the PMDCollab path (that bug was never about *which* origin the URL pointed to, it was about swapping to a blank placeholder before confirming the replacement image had actually loaded). `playGifArt()` mirrors `playPetAnim()`'s `new Image()` + `onload`-gated `reveal()` structure for this reason — don't copy `playStaticPoseArt()`'s no-preload shortcut here, that shortcut is only safe for `data:` URIs already resident in memory.

**Per-asset `holdMs` for one-shot (non-looping) animations.** Unlike `PET_ART_OVERRIDE`'s single shared `STATIC_POSE_MS` (fine for static poses with no "natural" duration of their own), a real GIF *has* a natural total loop duration (sum of its own per-frame delays) — cutting it off at an arbitrary shared constant risks stopping mid-loop at a visually awkward frame. `PET_GIF_OVERRIDE`'s per-state entries carry an optional `holdMs` (falls back to `DEFAULT_GIF_HOLD_MS` if omitted) set to roughly that GIF's own total duration — read via PIL (`im.info.get('duration', 100)` summed across `im.n_frames`), not guessed.

**Gaps where no matching asset exists for an existing state — ask, don't guess the mapping.** The Fuecoco set had no `Walk` GIF and one extra (`happy_jump.gif`) that didn't correspond to any existing state name. Two decisions were needed and were confirmed with the user via `AskUserQuestion` rather than assumed:
- **Missing state (Walk)**: reuse another provided asset (`idle.gif`) rather than silently falling through to PMDCollab's `Walk` sprite for that one state — falling through would recreate the exact idle/walk art-style mismatch the 2026-07-19 PMDCollab-unification fix (documented above) was written to solve, just for this one species instead of all of them. If unsure whether reuse vs. fallback is better for a future species, the rule of thumb is: **never let one visible-most-of-the-time state (Idle/Walk) silently diverge in art style from its sibling** — reuse *something* from the same asset batch even if imperfect, rather than mixing sources.
- **Extra/unmapped asset (happy_jump.gif)**: mapped onto the existing `Attack` key, since in this codebase `Attack` is *only* ever triggered as `interactPet()`'s 30% cosmetic reaction to petting the pet (see `interactPet()` — this is flavor, not tied to real battle move data) — a happy-jump reaction fits that trigger's actual meaning better than a punch/attack pose would have, and required zero changes to the trigger site itself since it's still keyed by the same `'Attack'` string.

**Don't assume a ready-made GIF faces right by default — check it.** The site-wide mirroring convention (`facingLeft` → `scaleX(-1)`, see the "Mirroring" note near the bottom of this file) was written against PMDCollab/AI-static art, both of which are drawn facing right by default. A ready-made `PET_GIF_OVERRIDE` asset has no such guarantee: Pikachu's `walk.gif` (皮卡丘, 2026-07-22 batch) is drawn facing/walking **left** by default — confirmed by extracting all frames with PIL (`ImageSequence.Iterator` + a contact-sheet PNG, not eyeballing the single-frame thumbnail) — so the old code showed Pikachu walking backwards from its facing direction (moving right on screen while the unflipped left-facing art displayed under `scaleX(1)`, and vice versa). Fixed by giving each `PET_GIF_OVERRIDE` entry an optional `facesLeft:true` flag and XOR-ing it against `facingLeft` in `applySpriteTransform()` (a separate `currentArtFacesLeft` module variable, set from `art.facesLeft` in `playGifArt()`'s reveal and reset to `false` in the PMDCollab/`playStaticPoseArt()` paths) rather than trusting movement direction alone to determine the flip. **When wiring up any future ready-made GIF for Walk (or any state where a viewer would notice directionality), pull a few frames with PIL first and actually look at which way the subject faces** — don't assume the "faces right by default" convention holds just because it did for every asset source used so far.

**Audit every place that assumes the old timing/sizing source when adding *any* new override path.** This pilot surfaced a real bug: `showFoodPop()` (the floating food emoji shown during feeding) computed its on-screen duration by reading `PET_ANIM[speciesId].Eat` frame-count × `FRAME_MS` directly — a calculation that's only valid for the PMDCollab path. For species 909 with the new override active, the real `Eat` animation played for ~1800ms (the GIF's own duration) but the food-pop icon vanished after ~720ms (the now-irrelevant PMDCollab-table math), visibly desyncing from the animation still playing underneath it. Fixed by extracting `getEatAnimDurationMs(speciesId)`, checked in the same priority order `playPetAnim()` itself uses (`PET_ART_OVERRIDE` → `PET_GIF_OVERRIDE` → `PET_ANIM` fallback) — but the general lesson is broader than this one call site: **anywhere code reads `PET_ANIM[speciesId][...]` directly to derive a duration, size, or other display fact (not through `playPetAnim()` itself), it's implicitly assuming the PMDCollab path is the one actually rendering — grep for other direct `PET_ANIM[` reads whenever a new override path is added, don't assume `playPetAnim()` is the only consumer.**

**Second bug this pilot surfaced, reported by the user right after shipping: the pet could walk off-screen.** `walkToRandomSpot()` (and `startStage()`'s initial placement) picks a random target and writes it straight to `img.style.left`/`top` — which anchors the **raw, pre-`transform:scale()`** box's top-left corner, not the visible box's. `transform:scale()` shrinks/grows the visible box around its *center* (default `transform-origin`), so raw-box-center and visible-box-center always coincide (which is why the existing `left = stageW/2 - w/2` centering math for Sleep etc. is fine untouched), but **edges** don't — the gap between the raw box's edge and the visible box's edge is `rawSize*(1-scale)/2`, negligible when raw≈rendered size (true of every asset before this pilot, PMDCollab and AI-static alike) but huge when they diverge (呆火鱷's raw 208-301px vs. its ~108px rendered size). The old code clamped the *raw* box's corner within a flat `margin=96` "safety net" that happened to be roughly the *rendered* size — coincidentally correct only because prior assets' raw and rendered sizes were close. With a much-larger-than-`ANIM_DISPLAY_TARGET` raw asset, that same margin left the raw box's center (and thus the whole visible, shrunk-down box) free to drift far outside the stage.

Fixed by clamping in **visual space**, not raw space: pick the random target as where the *rendered* box should sit within the stage bounds, then convert back to the raw `left`/`top` needed to place it there (`raw = visual - rawSize*(1-scale)/2`). Added `getAnimMeta(speciesId, animName)` — mirrors `playPetAnim()`'s own override-priority lookup — so both `walkToRandomSpot()` and `startStage()`'s initial placement compute this from the species' *actual* asset dimensions, not a hardcoded constant. **Any future edge-based positioning/collision math for `#pet-sprite` must go through this same visual-space conversion, not raw `style.left/top` directly** — this is the same category of mistake the companion-sprite `getBoundingClientRect()` gotcha (documented above) warns about, just for size instead of position: never assume a raw style property tells you where the pixels the user actually sees are, when a `transform` sits between the style and the render.

## Retired: wearable overlay landmark-finding (kept for reference only)

The pet-wearables feature (glasses/hats tracking eye position) was built, calibrated through three rounds of increasingly-correct fixes, and then **abandoned by explicit user request 2026-07-19** ("我放棄裝扮了") in favor of the companion sprite above — not because the methodology was unsound, but because pixel-perfect landmark tracking on a continuously-animating multi-frame GIF has an inherent precision ceiling that wasn't worth continuing to chase. All wearable code (`WEARABLE_SVG`, `WEAR_ANCHORS`, `WEARABLE_SLOTS`, shop wearable category, `renderWearLayers`/`syncWearPositions`) was removed from both `server.js` and `public/tamagotchi.html`.

If a future request revives per-landmark overlays (wearables, or something else that needs to track a specific point on the sprite), the methodology below is still correct and worth reusing rather than rediscovering:

1. **Calibrate against what's actually displayed, never a substitute.** `#pet-sprite`'s primary image is the *animated* GIF (PokeAPI Gen5 B/W or Showdown), not the static PNG fallback — these are different art assets at different native resolutions (Pikachu: static 96×96, animated GIF natively 50×46) stretched into the same 96×96 CSS box. The first wearables attempt calibrated against the static PNG and was wrong for all 13 species as a result; two rounds of "make the coordinates more precise" fixes didn't help because the underlying reference image was wrong, not the precision. Build the true reference by fetching the animated GIF, `seek(0)`, and resizing to exactly `(96,96)` with `Image.NEAREST` — measure coordinates on *that* image.
2. **Grid-overlay + crosshair-verify, don't eyeball.** Render the 96×96 reference at 10×+ zoom with a pixel grid every 8px, read the landmark position off the grid, then draw a crosshair at the candidate coordinate and re-view zoomed in before committing — this catches the multi-pixel errors that pure eyeballing misses.
3. **Mirroring**: `#pet-sprite` gets `scaleX(-1)` when the pet walks left (`facingLeft` variable, applied via `applySpriteTransform()`). Any anchor coordinate measured on the default right-facing image needs `mirroredAx = facingLeft ? (96 - ax) : ax` wherever it's used.
4. Automated dark-pixel clustering (to auto-detect eyes) was tried and produced too much noise — it picks up outline strokes as readily as pupils. Manual grid-reading with crosshair verification was more reliable for sprites this small.
